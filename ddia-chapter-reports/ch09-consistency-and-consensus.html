03 Consistency Guarantees: From Eventual to Linearizable

❌ Eventual Consistency

Definition: After writes cease and some unspecified time passes, all reads converge to same value.

Also called: Convergence, replicas eventually agree

❌ When convergence? ❌ Stale reads possible ❌ High developer burden

✅ Linearizability

Definition: System appears as single replica; all operations atomic; recency guarantee enforced.

Also called: atomic consistency, strong consistency, immediate consistency

✔ Immediate visibility ✔ Fresh reads guaranteed ✔ Simpler for apps

Isolation vs. Distributed Consistency

These are independent concerns that often get conflated:

Property Isolation (Transaction Level) Distributed Consistency (Replica Level)
Scope Race conditions from concurrent transactions Replica state coordination across delays & faults
Example Two txns read/modify same record simultaneously Write commits on node A; node B has stale value
Solution 2PL, SSI, MVCC within single database Replication lag elimination, total order broadcast
Relationship ⚠ Overlap but mostly orthogonal

04 Linearizability: Single-Replica Illusion

🎯 Core Idea

Maintain illusion of single copy of data where all operations are atomic and ordered by real time. Once a write completes, all subsequent reads see the new value.

Non-Linearizable Example (Figure 9-1: Football Score)

Replication Lag Violates Linearizability

Event Timeline:

T1: Game ends; replica A writes winner announcement

T2: Alice queries replica A → sees "Winner announced" ✓

T3: Alice tells Bob the result (out-of-band communication)

T4: Bob queries replica B (replication lag, update not yet arrived)

T5: Bob sees "Game still ongoing" ❌ VIOLATION

CAUSALITY VIOLATION Bob's query at T4 starts after Alice's read at T2 completes. By real-time causality, Bob should see at least as recent a value. Stale replica breaks this invariant.

Formal Operations & Conditions

Register operations (from DDIA §9.1):

  • read(x) ⇒ v — Read x, get value v
  • write(x, v) ⇒ r — Write x to v, return ok/error
  • cas(x, v_old, v_new) ⇒ r — Atomic compare-and-set; if x==v_old then set to v_new, else error

Three Critical Conditions

1

Read/Write Concurrency

Reads concurrent with writes may return either old OR new value. Reads before write starts → old value. Reads after write completes → new value. Overlapping reads → either acceptable.

2

No Value Flip-Flop

Once any read returns new value, ALL subsequent reads must return new value. Write flips value at single atomic point between write start & end. Enforces total order causality.

3

Total Order (Real-Time Ordering)

Each operation has vertical line in timeline. Operations form total order left→right (chronologically). Enforces recency: latest written value visible until overwritten. Reorderings impossible if causally related.

Allowed Reorderings

✔️
Requests Concurrent
Requests delivered out-of-send order if concurrent—linearizability permits arbitrary total order of concurrent ops
✔️
Response Delays
Response latency doesn't violate linearizability if operation processed in valid sequential order
❌
Causality Reversals
If A's read completes then B's query starts, B must see at least as fresh value as A. Reversing this order violates linearizability.

Verification: Formal Test (Figure 9-4)

Algorithm: Record all request/response timings → Check if arrangeable into valid sequential order that respects real-time causality.

⚠️ Computationally expensive (exponential in operation count), but provides definitive proof of linearizability or violation.

Linearizability vs. Serializability

Dimension Serializability Linearizability
Scope Multi-object transactions Single object (register)
Purpose Avoid race conditions between concurrent txns Guarantee recency + immediate visibility
Ordering Constraint Arbitrary serial order acceptable Must match real-time order
Prevents Write skew, race conditions Stale reads, replication lag
2PL ✔ Serializable AND Linearizable
SSI ✔ Serializable ✘ NOT Linearizable (uses consistent snapshots intentionally)

Use Cases: When Linearizability Matters

  • 🔐 Locking & Leader Election — Exactly one leader, all nodes agree; split brain prevention. Implementation: Treat lock as linearizable resource (ZooKeeper, etcd with consensus algorithms)
  • 🎯 Constraints & Uniqueness — Enforce unique email addresses, username exclusivity. Requires linearizable write-then-check pattern
  • ⚙️ Fencing Problem (Hidden Complexity) — Leader election alone insufficient; mechanism needed to prevent "zombie leader" from corrupting state (see p. 301 details)
  • 💻 Oracle RAC Example — Dedicated cluster interconnect + lock per disk page per node. Hardware-assisted linearizability.

The Core Problem: Asymmetric Failure

💥
Multi-Node Asymmetry
Transaction may commit on some nodes while failing on others—violating ACID atomicity guarantee. Example: Node A commits, network partition, Node B aborts.
🔄
Irrevocability Principle
Once committed, transaction is visible to other transactions. Retroactive abort violates read-committed isolation—other txns already read the committed data.
📝
Secondary Index Problem
Multi-object transactions must maintain consistency between primary data and secondary indexes (separate data structures). Partial commit breaks both.

Write-Ahead Log (WAL) Pattern

SINGLE-NODE MECHANISM Data written first to buffer, then commit record written to disk. Commit point is moment disk controller writes commit record. Recovery on restart: committed records are definite; missing = rolled back.

Multi-Node Challenge: Distributed Atomicity

Need: Guarantees across multiple independent database nodes that all either commit or all abort, despite any single node/network failure.

COMPENSATING TRANSACTIONS Effects can be undone via separate transaction, not retroactive abort. If Alice's balance decremented for transfer, and transfer fails at Bob's bank, Alice needs refund txn—not rollback.

Protocol Flow: Two Distinct Phases

1

Application Requests Global Transaction ID

Coordinator (library in app process or separate service) assigns universally unique global TX ID.

2

Single-Node Transactions Begin

Each participant starts local transaction with attached global TX ID. Working set accumulates on each node independently.

PHASE 1

Prepare Request (Canary Commit Check)

Coordinator sends prepare requests (tagged with global TX ID) to all participants. Each participant responds with "yes" = promise to definitely commit later OR "no" = abort immediately.

Critical Requirement: Participant voting "yes" surrenders right to abort. Must write all data to disk, check all constraints. No "disk space excuse" accepted—if you vote yes, you WILL commit.

5

Coordinator Writes Commit Decision (Irrevocable Point)

After receiving all prepare responses, coordinator writes commit decision record to its own transaction log on disk. This moment is irrevocable—system will commit regardless of future failures.

PHASE 2

Commit/Abort Messages (Must Retry Forever)

Coordinator sends commit or abort to all participants based on disk-written decision. Must retry forever on network failures—participant crashes, packet loss, timeouts. Participants execute decision when received.

On Participant Recovery: If voted "yes" → must commit when recovered. If not in coordinator log → abort. Coordinator log is source of truth.

System of Promises: The Atomicity Mechanism

🤝 Two-Part Guarantee

Promise 1 (Participant): "Yes" vote = commits later regardless of circumstances (hardware failure, power loss, network partition). Data durably persisted.

Promise 2 (Coordinator): Disk-written decision = irrevocable. Even if coordinator crashes, recovery will read log and replay commit/abort to all participants.

Single-node atomic commit lumps both into one event (write commit record to WAL). Distributed version requires two separate promises across the network.

The Blocking Problem: Coordinator Failure

IN-DOUBT STATE Participant votes "yes", then coordinator crashes before sending commit/abort request. Participant is stuck—cannot abort unilaterally (other participants may have committed), cannot commit unilaterally (unknown if others aborted).
🔒
Lock Holding During In-Doubt
Transaction holds row-level exclusive locks (for dirty writes) + shared locks (serializable isolation). Cannot release—would allow other txns to read potentially uncommitted state.
⏱️
Network Timeout Useless
Timeout doesn't distinguish coordinator crash from network failure. Can't safely abort because some participants may have committed.
⏳
Recovery Requirement
Participants must wait for coordinator to recover (read transaction log) to learn decision. Lock duration = in-doubt period (potentially 20+ minutes or indefinitely if log lost).
⚠️
2PC Blocking Issue: If coordinator fails after participants vote "yes", locks are held indefinitely until coordinator recovers and replays decision. In-doubt period can be very long, blocking other transactions and degrading throughput. This is why 2PC has fallen out of favor in modern architectures.

07 FLP Impossibility & Practical Consensus Algorithms

📜 FLP Result [Fischer, Lynch, Paterson]

Theorem: No algorithm guarantees consensus if a single node crashes in an asynchronous system model with deterministic algorithms (no clocks, timeouts, randomization).

This is not just a limitation of current algorithms—it's a fundamental impossibility result for the model.

Practical Bypasses to FLP

⏱️ Timeouts & Failure Detection

Assume nodes are "down" if no response within timeout window. Breaks asynchrony assumption (introduces timing) and enables consensus.

  • Used by Raft, Paxos, Zookeeper
  • Risk: False positives (slow node falsely suspected)
  • Pragmatic trade-off for real systems

🎲 Randomization

Probabilistic algorithms (random exponential backoff, random timeouts) sufficient to bypass FLP impossibility.

  • Randomness breaks determinism requirement
  • Enables liveness with fault tolerance
  • Used in Raft (random election timeout)

2PC as a Consensus Algorithm (Problematic)

2PC is reducible to/from consensus (references [70, 71]) but is not a good consensus algorithm for these reasons:

Aspect 2PC Limitation Why It Matters
Blocking ✘ Blocks on coordinator failure Locks held indefinitely; impacts system availability
Participants Required ✘ Requires all participants responsive Single slow/down node blocks entire system
Recovery Process ✘ Weak recovery guarantees In-doubt state can persist for days if log lost
Fault Tolerance ✘ Single coordinator SPOF No replication of
