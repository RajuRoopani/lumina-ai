Historical Milestones (1890–2004)

1890 — Hollerith Machines (US Census)

Automated census tabulation using punch cards and sorting machines — first large-scale data processing system.

1940s–1950s — IBM Card Sorting

Specialized hardware for batch processing of punch-card datasets; became backbone of business data processing.

1978 — Unix Philosophy Codified

McIlroy's four principles for composable text-processing tools — foundation of modern batch pipelines.

2004 — MapReduce Paper (Google)

Distributed batch processing framework for thousands of machines; opened era of datacenter-scale offline jobs.

03 Unix Philosophy & Design Principles

🎯
Principle 1: Single Responsibility
Each program does one thing well; build fresh rather than complicate existing tools.
🔗
Principle 2: Universal Interface
Output becomes input to unknown programs; avoid stringent formats.
⚡
Principle 3: Rapid Iteration
Iterate quickly; rebuild clumsy parts rather than plan perfectly upfront.
🛠️
Principle 4: Tools Over Manual Work
Use tools; expect to throw tools away as needs evolve.

Uniform Interface: Files as Byte Sequences

The fundamental abstraction: everything is a file — stdin/stdout/filesystem.

  • 🔵 Inversion of Control Programs read stdin, write stdout; shell controls wiring. Program doesn't know source or destination.

04 Unix Tools & Log Analysis Pipelines

🔍 Real-World Example: Web Access Log Analysis

Extract top 5 most-accessed URLs from nginx access log — demonstrating field extraction, sorting, aggregation, and ranking.

Pipeline Decomposition

cat /var/log/nginx/access.log | awk '{print $7}' | sort | uniq -c | sort -r -n | head -n 5
1

cat — Stream Input

Read entire file sequentially; pipe bytes to next stage. Handles gigabytes via OS disk buffering.

2

awk — Field Extraction

Parse each line; extract 7th whitespace-separated field (URL). Per-record state, emits one output per input.

3

sort | uniq -c — Grouping & Aggregation

Sort by URL (enables grouping); `uniq -c` counts consecutive identical lines. Spills to disk if dataset > RAM.

4

sort -r -n — Ranking

Reverse numeric sort on counts; brings highest-count URLs to top.

5

head -n 5 — Top-K Selection

Output only first 5 lines; terminates, stopping upstream processing (important efficiency pattern).

GNU Coreutils `sort` Optimization Automatically spills to disk when data exceeds available RAM, parallelizes across CPU cores, and achieves throughput bounded only by disk read speed. Demonstrates that even simple tools benefit from careful implementation for large-scale data.

Scalability & Performance

  • ⚡ Typical Scale Processes gigabytes of logs in seconds; throughput-limited by disk read bandwidth (~100–500 MB/s on commodity disks).
  • 📊 Disk Spillover `sort` checks available RAM and automatically spills intermediate results to temporary files, enabling processing of datasets far larger than memory.
  • 🔄 Parallelization Modern `sort` spawns worker threads per CPU core; I/O bound for most large datasets.

HDFS: Shared-Nothing Distributed Storage

  • 🔵 NameNode (Central Metadata) Tracks file-to-block mapping and block-to-machine distribution; single point of metadata, not data.
  • 🔵 Data Nodes (Compute Machines) Each machine runs daemon exposing local block access via network; files striped into blocks (~128 MB/block default).
  • 🟢 Block Replication (3x Default) Each block replicated across 3 machines for fault tolerance; lost machine doesn't lose data.
  • 🟣 Reed-Solomon Erasure Coding Alternative to full replication; reconstructs data from fewer blocks using parity — lower overhead for archival data.
  • 🟢 Commodity Hardware + Open Source Drives dramatically lower cost vs. dedicated storage appliances (NAS/SAN); scales to petabytes.

Key Design Principles

✅ Shared-Nothing Strengths

  • No network congestion from centralized storage
  • Each machine independent; one failure doesn't cascade
  • Scales horizontally: add machines, get capacity
  • Enables **computation near data** (reduces network)

❌ Shared-Disk (NAS/SAN) Trade-offs

  • Centralized bottleneck; network can become saturated
  • Storage appliance is single point of failure (mitigated via RAID, but still centralized)
  • Cannot scale cost-effectively to petabytes
  • Computation runs on compute nodes, must fetch data over network
Write-Once Semantics HDFS enforces write-once, sequential semantics (no modification post-write) — aligned with immutability principle. Enables simple consistency model and reproducible job runs.

06 MapReduce Job Execution Flow

1

Read Input Files, Break into Records

Framework reads input directory from HDFS; splits files/blocks into records using delimiter (newline by default). Each mapper task processes one contiguous chunk of input.

2

Map Phase — Stateless Key-Value Extraction

Mapper called once per input record; examines record, emits 0 or more key-value pairs. No state retained across records; each record processed independently. Analogous to `awk` field extraction.

3

Implicit Shuffle & Sort by Key

Framework collects all key-value pairs, sorts by key, and partitions into groups. Enables next stage (reduce) to iterate over values for each key in sorted order.

4

Reduce Phase — Stateful Aggregation

Reducer called once per unique key; receives iterator over all values for that key (already sorted). Can accumulate state across values (sum, count, etc.) and emit final output.

Distributed Execution: Parallelization Strategy

Partitioning & Locality Optimization

  • 🔵 Input = Directory Each file or block becomes one map task; if input is 100 GB with 128 MB blocks, creates ~800 map tasks.
  • 🟢 Computation Near Data Scheduler tries to place map task on machine storing input block replica. Saves network bandwidth; critical optimization at scale.
  • 🟠 Data Locality Fallback If machine is busy, scheduler picks nearest replica (same rack, then different rack). Balances locality with load.
  • 🟣 Partition by Key Hash Framework partitions intermediate key-value pairs into R reduce tasks using hash(key) % R, distributing work across reducers.
💡
MapReduce Parallelism: Input splits drive map parallelism; hash partitioning drives reduce parallelism. Each mapper runs independently on a single block (or portion thereof); shuffle/sort happens between map and reduce, coordinated by framework.

07 Materialization of Intermediate State

⏸️
Sequential Execution Bottleneck
Jobs must wait for ALL tasks in preceding job to complete. Stragglers (slow tasks) delay entire workflow — classic waterfall dependency.
🔄
Redundant Mappers
Mappers read back reducer output (already partitioned/sorted), re-partition and re-sort identically. Wasted computation if output already correctly partitioned.
💾
Over-Replication
Intermediate data replicated 3x in HDFS (like input data), but temporary. Wastes disk space and I/O for data that doesn't require long-term durability.

I/O Cost of Materialization Each MapReduce job writes intermediate results to disk (sorted, partitioned), then next job reads them back. This is safe and recoverable but incurs heavy I/O relative to computation for many workflows — key motivation for dataflow engines.

Cost-Benefit Analysis: When to Materialize

Intermediate data << Source data (e.g., 10% after filtering) Materialize
Computation is CPU-intensive (sort >> I/O of recompute) Recompute
