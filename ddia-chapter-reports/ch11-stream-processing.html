02 Stream Processing Fundamentals

  • 🌊 StreamUnbounded data made available incrementally over time; counterpart to batch data with unknown/infinite size. Cannot be sorted; restart infeasible after years of operation.
  • 📍 EventSelf-contained, immutable record containing details of something that happened at a specific point in time, typically with a timestamp. Single unit of stream.
  • 📤 Producer/Publisher/SenderEntity that generates events once and makes them available to the system. Single-shot responsibility.
  • 📥 Consumer/Subscriber/RecipientEntity that processes events, potentially multiple times. May apply logic, enrichment, or aggregation.
  • 🏷️ Topic/StreamLogical grouping of related events; organizational unit for subscription and publishing.
  • ⚙️ Operator/JobCode that processes input streams to produce derived streams; transformation or stateful aggregation unit.

🔑 Stream vs. Batch: Fundamental Distinction

Batch: Bounded input of known/finite size; entire dataset available; sorting possible; restart from beginning feasible.

Stream: Unbounded input arriving gradually; no complete dataset ever available; sorting impossible; continuous processing required; restart from beginning infeasible after extended operation.

03 Messaging Systems Architecture

Producer-Consumer System Design Questions

Q1: Speed Mismatch

Producers emit faster than consumers can process. Three responses: drop messages (data loss), buffer in queue (memory/disk overhead), or apply backpressure/flow control (block producer until capacity available).

Q2: Durability & Loss

Node crashes cause message loss unless durable. Write to disk + replication required. Context determines tolerance: sensor readings/metrics accept gaps; event counting cannot lose any message (breaks correctness).

Design Decision: Backpressure Mechanisms

Unix pipes and TCP both use fixed-size buffering with backpressure. Producer blocks when buffer full. Trade-off: latency (backpressure waits) vs. data loss (drop) vs. memory explosion (unbounded buffering).

04 Messaging System Architecture Layers

Direct Messaging vs. Message Brokers

Direct Messaging (UDP, ZeroMQ, Webhooks)

  • Producer sends directly to consumer endpoint (multicast, TCP, HTTP callback)
  • No central broker; no durability by default
  • Latency low; fault tolerance poor (offline consumer misses events)
  • Examples: UDP multicast (stock feeds), ZeroMQ/nanomsg, StatsD metrics, webhooks

Message Brokers (Centralized Server)

  • Server acts as intermediary; producers/consumers are clients
  • Broker centralizes data; tolerates client churn and offline periods
  • Asynchronous: producer confirmed when broker receives, not when consumer processes
  • Durability configurable (memory vs. disk-backed); unbounded queueing typical
  • Examples: RabbitMQ, ActiveMQ, HornetQ, TIBCO MQ, Google Cloud Pub/Sub
BROKER ARCHITECTURE ADVANTAGE Unlike direct messaging, brokers decouple producer and consumer lifetimes. Producer completes after broker confirmation (not consumer confirmation). Broker buffers events if consumer offline. Enables flexible topology changes and fault isolation.

Multi-Consumer Patterns

Load Balancing

Each message delivered to exactly one consumer. Broker assigns arbitrarily (round-robin or queue binding). Enables parallel processing of expensive messages. Consumers unaware of each other.

AMQP: multiple clients on same queue JMS: shared subscription

Fan-out

Each message delivered to all subscribers independently. Independent consumers without mutual effect. Reading message does not delete it; each consumer sees full stream.

JMS: topic subscriptions AMQP: exchange bindings

Acknowledgments & Redelivery

1

Consumer Processes Event

Client receives message and begins processing (may take seconds, minutes, or longer).

2

Explicit Acknowledgment

Client signals completion to broker when safe to forget message. Broker removes from queue only after ACK.

3

Failure: Redelivery

If connection closes or timeout occurs without ACK, broker redelivers message to another (or same) consumer. Message visible again in queue.

4

Lost ACK Problem

ACK message itself can be lost between client and broker, causing duplicate delivery. Requires atomic commit protocol to fully eliminate.

⚠️
Redelivery Causes Reordering
Load balancing + redelivery = inevitable message reordering. Example: Consumer 2 crashes on message m3 while Consumer 1 processes m4; m3 redelivered to Consumer 1 → order becomes [m4, m3, m5] not [m3, m4, m5]. Mitigation: one queue per consumer (disables load balancing).
🔍
Ordering vs. Parallelism Trade-off
Strict ordering requires single consumer per queue. Multiple consumers enable parallelism but break message sequence. Acceptable if events independent; problematic with causal dependencies.

CORE INSIGHT Partitioned logs combine database durability (append-only disk storage) with messaging efficiency (low-latency streaming). Writes append to disk sequentially; reads can tail the log like Unix tail -f. Replication across brokers provides fault tolerance.

Key Advantages Over Traditional Brokers

  • 💾 Built-in DurabilitySequential disk append is efficient (no random I/O); log persists indefinitely; consumers can replay historical events.
  • 🔁 ReplayabilityConsumers store offset position; can rewind and reprocess from any point. Enables bug fixes (replay with corrected logic) or new consumer additions.
  • 📊 Partitioning for ScaleEach topic divided into independent partitions; each partition assigned to one broker. Enables horizontal scaling and parallel consumption within topic.
  • 🌊 Consumer FlexibilityConsumers pull at own pace (not pushed); broker doesn't need to track consumer state beyond offset; easier topology changes.

07 Timestamps & Clock Synchronization

🚨
Clock Untrustworthiness
Device clocks (phones, IoT, personal computers) are notoriously inaccurate. Users can set them incorrectly; NTP sync is unreliable; clocks drift. Server clocks more reliable but semantically less meaningful (user action timestamps).
⏱️
Mobile Offline Buffering
Device offline for hours/days, then reconnects and sends buffered events. To server, all events appear hours/days late (extreme stragglers). Naive processing assumes current time = real time, causing incorrect windowing and temporal analysis.
💡
Semantic vs. Technical Trade-off
Device time = user action time (semantically correct but untrustworthy). Server time = controlled/accurate but displaced from real user action moment. Must choose which timestamp matters for business logic.

Three-Timestamp Solution

Log three distinct timestamps per event to triangulate true event occurrence time:

Timestamp 1: Event Occurrence Time Device clock at moment of event (user interaction)
Timestamp 2: Event Send Time Device clock when event transmitted to server (may be much later if offline)
Timestamp 3: Event Receipt Time Server clock when message arrives (always trustworthy)

Offset Estimation Formula

offset = server_receipt_time - device_send_time
// Assumes negligible network delay (usually reasonable for cloud)

true_event_time ≈ device_event_time + offset
CRITICAL CAVEAT This formula assumes device clock offset remained constant between event occurrence and send time. If device clock jumped (user corrected time, DST change), formula fails. No perfect solution; this is best-effort approximation.

08 Windowing Types for Stream Aggregation

Tumbling Window

Fixed-length, non-overlapping intervals. Every event belongs to exactly one window. Simplest aggregation.

10:03:00 – 10:03:59 [events...] ✔
10:04:00 – 10:04:59 [events...] ✔
No overlap

Implementation: window = round_down(timestamp, window_size)

Hopping Window

Fixed-length, overlapping windows. Provides smoothing across multiple adjacent windows.

10:03:00 – 10:07:59 [events...] ✔
10:04:00 – 10:08:59 [events...] ✔
1-min hop, 5-min window

Use case: smoothed metric dashboards, trend analysis

Sliding Window

Captures all events within time interval of each other (not fixed boundaries). More continuous but higher state overhead.

Event @ 10:03:39
Event @ 10:08:12
5-min apart ✔ (match!)
Tumbling/hopping would NOT match

Implementation: sorted buffer, expire old events

★ SESSION BASED

Session Window

No fixed duration. Groups events for same entity (user) occurring closely together. Ends on inactivity timeout.

User 42: 10:00–10:35
[inactivity 30+ min]
User 42: 11:20–11:45
Two sessions, not one

Use case: website analytics, user session tracking

Windowing Trade-offs Summary

State Overhead Sliding > Hopping > Tumbling > Session
Latency (Result Delay) Tumbling > Hopping > Sliding ≈ Session
Smoothing Quality Sliding > Hopping > Tumbling
