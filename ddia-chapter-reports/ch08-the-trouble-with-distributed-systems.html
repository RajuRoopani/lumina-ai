02Fault Categories

💥
Partial Failures
Definition: System components fail independently while others continue operating, creating nondeterministic outcomes. The fundamental challenge of distributed systems.
⏱️
Indistinguishability Problem
Network gives no information about arrival time or delivery confirmation. Delays ≈ failures at the sender. Only success evidence: positive application response.
🎲
Nondeterminism
Operations may succeed, fail, or remain indeterminate. Same request may work once and fail later. Software must handle: wrong assumptions about state.

Specific Failure Modes in the Wild

  • 🔵 Network Transient GlitchesPacket drops, reordering, delays—especially in public cloud (EC2) and datacenter upgrades. Switch software upgrade caused >1 minute packet delays.
  • 🟠 Asymmetric Link FailuresOne-directional connectivity loss: inbound packets accepted, outbound dropped. Detected only by application layer timeout.
  • 🔴 Infrastructure FailuresRack disconnections (~50% of medium datacenter faults), undersea cable cuts, router misconfigurations (majority cause of redundant gear faults).
  • 🟣 Garbage Collection PausesRemote node temporarily unresponsive while GC runs—indistinguishable from crash to sender.

03Single-Node vs. Distributed System Design Philosophy

Design Philosophy Comparison
Dimension Single-Node Systems Distributed Systems
Determinism Same operation always produces identical results Nondeterministic outcomes; must handle uncertainty
Error Handling Prefer total crash (kernel panic) over silent wrong results Accept partial failures; service must remain available
Failure Model Idealized system model with mathematical perfection Confront messy reality; no shared model between nodes
Recovery System crash = restart; clear state transition Rolling upgrades, VM replacement, geographic distribution

HPC (Supercomputers) vs. Cloud Computing Paradigms

🖥️ HPC Supercomputers

Failure Philosophy: Escalate partial failures → total failure

  • Checkpoint & restart on any node failure
  • Specialized high-reliability hardware (RDMA, shared memory)
  • Mesh/torus network topologies optimized for throughput
  • Acceptable for batch jobs (offline computation)

☁️ Cloud Computing

Failure Philosophy: Tolerate partial failures; maintain availability

  • Service remains online despite component failures
  • Commodity hardware; higher failure rates expected
  • IP/Ethernet Clos topologies; packet networks
  • Mandatory for online services requiring low latency
SCALING IMPLICATION System with thousands of nodes: reasonable to assume something is always broken. The benefit of fault tolerance is realized through rolling upgrades, VM replacement, and geographic data distribution (Chapter 4).

Proven Approaches

  • 🟢 Error-Correcting CodesDetect and correct bit errors in unreliable communication channels—proven to work for decades in telecommunications.
  • 🟢 TCP over Unreliable IPIP network provides no delivery guarantees (drops, delays, duplicates, reordering); TCP adds retransmission, duplicate elimination, and packet reordering logic on top.
  • 🟢 Redundancy & ReplicationMultiple copies ensure data survives single-component failures; quorum reads/writes ensure consistency.

🏆 Core Principle: Higher-Level Systems Hide Low-Level Faults

Once low-level faults are masked, remaining faults at application level become easier to reason about and handle. System reliability emerges from layered abstractions that progressively isolate higher layers from physical unreliability.

Fundamental Limits

CANNOT REMOVE DELAYS Reliability construction cannot magically eliminate network latencies. We can retry, duplicate-check, and reorder—but the inherent unpredictability of packet arrival times remains. This bounds achievable system performance.

06Six Failure Modes: The Indistinguishability Problem

Core problem: When you send a request and receive no response, what actually happened? All six scenarios are indistinguishable at the sender.

1️⃣
Request Lost
Network dropped your outgoing packet before reaching recipient.
2️⃣
Network Overloaded
Request queued inside network or recipient's NIC; delayed indefinitely.
3️⃣
Remote Node Crashed
Recipient failed; OS will not respond at all.
4️⃣
Temporary Pause (GC)
Remote node paused by garbage collection or OS scheduling; behaves exactly like slow network.
5️⃣
Response Lost
Remote processed request successfully; response packet dropped on return path.
6️⃣
Response Delayed
Response queued in network; arbitrary delay before arrival.

Critical Insight: TCP Acknowledgment is Insufficient

TCP ACK ≠ APPLICATION SUCCESS TCP acknowledgment only confirms that the OS kernel received bytes on the remote machine. The application could crash immediately after TCP ack, losing the request. Only evidence of success: positive response from the application itself, fully processed and returned to you.
💡
No Magic Detection: The sender must assume no response could mean any of six scenarios. No protocol addition eliminates this fundamental ambiguity without central clock synchronization (which is itself unreliable).

Faults from Human Misconfiguration

Real Failure Examples from the Field

  • 🔴 Switch Software UpgradeTopology reconfiguration caused >1 minute packet delay window. During this window, timeouts and retries cascade, triggering cascading failures in dependent services.
  • 🔴 Undersea Cable CutsSharks and ship anchors physically sever submarine cables. Geographic redundancy (multiple routes, different cables) essential for global services.
  • 🟠 Asymmetric Link FailuresOne-directional connectivity: inbound packets accepted, outbound dropped. Detected only when application timeout fires; no explicit OS error.
  • 🟠 Public Cloud GlitchesEC2 and similar platforms experience frequent transient network glitches; small percentage of requests fail mysteriously.
  • 🟠 Well-Managed Private DatacentersMore stable than cloud but not immune. Redundant networking gear doesn't proportionally reduce faults—human misconfiguration is the major cause.

🏆 Why Redundancy Often Fails

Redundant networking equipment (dual switches, multiple cables) does not proportionally reduce faults because human misconfiguration is a major fault source. If both redundant paths are misconfigured identically, redundancy provides zero protection. Requires independent configuration verification and testing.

08Detecting Network Faults: Explicit Mechanisms & Limitations

Available Detection Mechanisms (All Unreliable)

1

TCP RST / FIN (OS-Level)

Operating system closes connection if no process is listening on the remote port. Limitation: Provides no information about partial request processing; only confirms kernel rejection.

2

Crash Notification Script (Application-Level)

HBase and similar systems notify other nodes before timeout occurs. Limitation: Requires special implementation; not universally available. GC pauses or slow kernel scheduling still trigger false alarms.

3

Management Interface Queries (Infrastructure)

Network switches provide management ports for status queries. Limitation: Unavailable in internet services; impractical in shared datacenters (EC2, GCP, Azure).

4

ICMP Destination Unreachable (Router Feedback)

Router may respond if IP address unreachable. Limitation: No magic capability; same indistinguishability applies. Message can be lost or blocked by firewalls.

FUNDAMENTAL CONSTRAINT Rapid fault feedback is useful but never reliable. Must assume worst case: no response at all. Timeout-based detection remains the most practical approach, but timeouts are inherently difficult to tune (too short = false positives; too long = slow recovery).

Absence of Error Handling = Catastrophe

  • 🔴 Untested Fault HandlingWhen software encounters unanticipated failures, it does "arbitrary unexpected things"—permanent deadlocks, state corruption, data deletion.
  • 🟠 Cascading FailuresOne component's error handling triggers errors in dependent systems; failures propagate outward.
  • 🟢 Mitigation StrategyDeliberately trigger network faults in testing (Chaos Monkey approach). Know exactly how software behaves under specific fault scenarios. Ensure recovery is always possible.
💡
Not about tolerating faults: It's acceptable to show error to users during infrastructure problems. Essential: Know how software reacts under partial failure; ensure operators can recover. Test fault handling continuously.

Byzantine Fault Definition & History

BYZANTINE FAULT Node deliberately sends arbitrary, faulty, or corrupted responses. May claim receipt of messages it never received. Different from crash faults (stops responding) or transient faults (delayed response). Named after Lamport et al.'s "Byzantine Generals Problem" (1982)—a diplomatic choice to avoid offending any nation.

Byzantine Generals Problem

Setup: n generals must agree on battle plan. k generals are traitors who may send false/deceptive messages. Loyal generals send truthful messages. Traitors unknown a priori. Requires supermajority (>2/3) honest nodes to reach consensus on correct action.

SUPERMAJORITY REQUIREMENT System with 4 nodes can tolerate at most 1 Byzantine fault (>2/3 = 3 honest). With 7 nodes: 2 faults tolerated. Algorithm overhead grows substantially with required fault tolerance level.

When Is Byzantine FT Actually Needed?

✅ Relevant Domains

  • Aerospace: Radiation-induced memory/CPU corruption → unpredictable node behavior
  • Blockchain/P2P: Mutually untrusting parties; consensus without central authority
  • ISS / Aircraft Systems: Failure cost extremely high (collisions, crashes)

❌ Impractical for Datacenters

  • Organizational Trust: Cloud infrastructure under single organization's control
  • Low Corruption Rates: Radiation, bit flips rare in well-managed datacenters
  • Protocol Complexity: Algorithms extremely complex; high deployment cost
  • Same-Software Vulnerability: Bug in one node = bug in all if same software deployed

Software Bug Vulnerability

CANNOT PROTECT AGAINST BUGS Byzantine FT cannot protect against identical bugs deployed to all nodes. Would require 4+ independent implementations per bug tolerance. Equally ineffective against security vulnerabilities—if one node compromised, likely all are due to shared software/configuration.

Weak Forms of Lying: Pragmatic Protections

Instead of full Byzantine FT, defend against invalid messages from hardware, software, or misconfiguration:

  • 🔵 Network Corruption DetectionHardware/driver/OS/router bugs corrupt packets. TCP/UDP checksums catch most; some evade detection. Mitigation: Application-level protocol checksums (HMAC, CRC over entire message).
  • 🔵 Input SanitizationPublicly accessible apps require strict input checks (range validation, string size limits, format validation). Internal services (firewall-protected) can use lighter checks. All systems: Basic sanity-checking in protocol parsing.
  • 🔵 NTP Multi-Server StrategyClient contacts multiple time servers; estimates their errors; requires majority agreement on time range. Misconfigured outlier detected and excluded from sync. Result: Robustness exceeds single-server approach without full Byzantine complexity.
💡
Client-Side Untrust (Web Apps): End-users control browsers (expected to be arbitrary/malicious). Defense: Input validation, sanitization, output escaping (prevent SQL injection, XSS). Not used: Byzantine fault-tolerant protocols—instead rely on server as authority for allowed behavior.

11Algorithm Correctness: Safety & Liveness Properties

Goal: Formalize what "correct" means for distributed algorithms using safety and liveness properties. Enables rigorous reasoning about algorithm behavior under failures.

Core Property Definitions

Property Type Definition Violation Nature Recovery
Safety "Nothing bad happens"
Invariant that must never be violated		 ✔ Identifiable
Violated at specific point in time		 Cannot be recovered; system corrupted if violated
Liveness "Something good eventually happens"
Desired outcome must occur within bounded time		 ⚠ Asymptotic
Never violated at finite time; only if waited forever		 Wait longer or fix system state; no corruption if violated temporarily

Fencing Token Example: Formal Properties

✓

Uniqueness (Safety)

No two requests to lock service return the same token. If request X returns token tₓ before request Y returns token tᵧ, then tₓ < tᵧ. Violation = lock safety broken.

✓

Monotonic Sequence (Safety)

Tokens issued in strictly increasing order. Once token N issued, token N-1 can never be issued later. Prevents: duplicate token issuance, token reuse, sequence gaps creating confusion.

✓

Availability (Liveness)

Non-crashing node eventually receives response to lock request. May be delayed arbitrarily; liveness allows unbounded wait, but guarantees eventual completion under crash-recovery model.

Safety vs. Liveness: Practical Implications

🔐 Safety Properties

  • Cannot be recovered: Violation = permanent system corruption
  • Must hold even during failures: Protection during crash-recovery, Byzantine attacks
  • Examples: Uniqueness, consistency, atomicity, correctness invariants
  • Testing strategy: Exhaustive state exploration; formal proofs

⏱️ Liveness Properties

  • Temporary violations acceptable: Delays don't corrupt data; just slow response
  • Harder to guarantee: Requires system eventually recovers after failures
  • Examples: Availability, termination, eventual consistency
  • Testing strategy: Timeout simulation; chaos engineering; recovery verification
DESIGN TRADE-OFF Systems can sacrifice liveness to guarantee safety. Example: Distributed locking that doesn't return response during network partition (violates liveness) but guarantees only one client holds lock (maintains safety). This is often correct trade-off for critical systems.

Atomic Read-Modify-Write in Presence of Failures

Challenge: Compare-and-swap (CAS) across multiple nodes requires atomic visibility. Network partition or crash mid-operation creates ambiguity: did update occur? System must define semantics under partial failure.

  • 🔵 Linearizability (Strong Consistency)All operations appear to execute in single-threaded order. Safety property. Sacrifices availability during network partition (CP from CAP theorem).
  • 🔵 Eventual ConsistencyLiveness property. Read may return stale data during partition; eventually converges to same value. Maintains availability at cost of temporary inconsistency.
  • 🔵 Read-Your-WritesHybrid property. Your own writes always visible; others' writes eventually consistent. Common in distributed databases (DynamoDB).
💡
Safety & Liveness Determine Algorithm Design: Safety requirements (uniqueness, atomicity, correctness) must be non-negotiable. Liveness requirements (availability, latency) determine acceptable trade-offs during failures. Explicit property definition enables rigorous algorithm reasoning.

Cascading Design Implications

1

Acknowledge Partial Failures Are Possible

Unlike single-node systems (crash completely), distributed systems can have some components broken. This nondeterminism is your starting assumption.

2

Accept Network Indistinguishability

You cannot distinguish delay from failure. TCP ack insufficient. Timeout-based detection is inherently imperfect. Plan for false positives and false negatives.

3

Choose Your System Model

Synchronous? Partially synchronous? With crash-stop or crash-recovery faults? This choice constrains what algorithms are possible and what guarantees you can make.

4

Define Safety & Liveness Requirements

What must never happen (safety)? What must eventually happen (liveness)? Trading availability for consistency is explicit trade-off between these properties.

5

Implement Fault Tolerance Mechanisms

Retries, timeouts, error-correcting codes, quorum reads/writes, replication. Each addresses specific failure modes; combinations create robust systems.

6

Test Relentlessly Under Failure Scenarios

Chaos monkey. Network fault injection. Byzantine node simulation. Verify software behavior under each failure mode. Ensure operators can recover.

Why Distributed Systems Are Hard

  • 🔴 No Shared State or ClockNodes cannot directly observe each other's state. No central truth about time. Coordination requires message passing (fallible network).
  • 🔴 Unbounded DelaysNetwork/process pauses may exceed any chosen timeout. Causes cascading retries, load spikes, false failure detection.
  • 🔴 Asynchronous ExecutionNo global ordering of events. Causality tracking requires explicit mechanisms (vector clocks, timestamps).
  • 🔴 Fault/Delay IndistinguishabilityCore ambiguity: is node down or just slow? Sender cannot know without application-level confirmation.
  • 🟠 Byzantine Scenarios Rare but PossibleMost cloud systems assume organizational trust (not Byzantine). But still must defend against bugs, bit flips, network corruption. Weak-form defenses (checksums, input validation) essential.

13Practical Mitigation Strategies for Distributed System Faults

Core Strategy: Embrace Failure as Expected State

🏆 Design Principle: Build Expecting Failure

Do not treat failures as anomalies. Design for partial failures as the normal operating mode. Infrastructure should remain online and serve users (perhaps with degraded experience) even when components break. This shift in mindset determines architecture choices.

Proven Mitigation Patterns

★ ESSENTIAL

Timeouts & Retries with Exponential Backoff

When no response received, retry after increasing delay (1s, 2s, 4s, 8s...). Prevents thundering herd. Must be idempotent (safe to retry multiple times without changing outcome).

Reduces cascading failures Recovers from transient glitches Adds latency Requires idempotency

Replication & Quorum Reads/Writes

Multiple copies ensure data survives single-component failures. Quorum (majority) ensures consistency: write to n/2 + 1 replicas; read from n/2 + 1 replicas. Minority node failures tolerated.

Survives node failures Increases read parallelism Complexity in coordination Network partition splits brain

Bulkheads & Circuit Breakers

Isolate failed components: don't continue hammering dead service. Circuit breaker opens after threshold failures; stops requests temporarily. Prevents cascading failures from propagating. Bulkheads isolate resources (thread pools, connection pools) per service.

Prevents cascade Graceful degradation Requires explicit configuration May miss brief recoveries

Health Checks & Proactive Replacement

Periodically verify node health. Remove unhealthy nodes from serving traffic. VMs can be reprovisioned automatically. Prevents cascading requests to sick components.

Early detection Reduces failed request ratio False positives trigger needless replacements Overhead of health checks

Graceful Shutdown & Draining

Before restarting node (rolling upgrade), stop accepting new requests. Wait for in-flight requests to complete. Prevents data loss and client errors. Drain connection pools cleanly.

Zero request loss during restarts Improves rolling upgrade reliability Adds latency to drain phase Requires load balancer coordination

CHAOS MONKEY PHILOSOPHY Systematically break things in production-like environments to discover failure modes before users encounter them. Netflix pioneered this: randomly kill production instances to force teams to design fault tolerance. What you don't test fails in production.

14Key Takeaways & Design Principles

💡
Partial Failures Define Distributed Systems: Single-node systems either work completely or crash completely (deterministic). Distributed systems have partial failures—some components broken, others work. This nondeterminism is the core challenge requiring explicit fault tolerance design.
💡
Network Indistinguishability Is Fundamental: You cannot distinguish a failed node from a slow network using timeouts alone. TCP ack insufficient. Only evidence of success: positive application-layer response. Embrace this ambiguity; design around it with retries, idempotence, and deduplication.
💡
System Models Enable Reasoning: Synchronous, partially synchronous, asynchronous; crash-stop, crash-recovery, Byzantine. Choose your model explicitly. It determines what algorithms are possible and what guarantees you can make. Partially synchronous + crash-recovery most reflects reality.
💡
Safety vs. Liveness Is an Explicit Trade-off: Safety (nothing bad happens) cannot be violated without data corruption. Liveness (something good eventually happens) can be temporarily sacrificed for consistency. CAP theorem formalizes this: choose CP (consistency + partition tolerance) or AP (availability + partition tolerance) based on requirements.
💡
Byzantine Faults Rarely Needed in Cloud: Organizational infrastructure assumes honest-but-fallible nodes (crash-recovery model). Full Byzantine FT impractical. Defend pragmatically: checksums for corruption detection, input validation for invalid messages, multi-server strategies (NTP) to reject outliers.
💡
Expect Failures; Test Relentlessly: Chaos engineering discovers actual failure modes. Network fault injection, node crashes, clock skew—systematically break things to verify graceful degradation. What you don't test fails at scale. Operators must understand how software behaves under specific fault scenarios.
