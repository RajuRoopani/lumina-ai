Key Design Implication

Cloud systems MUST tolerate failed nodes. Rolling upgrades, VM replacement, and geo-distribution require fault-tolerance baked into software architecture—not optional, fundamental.

PARADIGM SHIFT You cannot design distributed systems with single-machine assumptions. Nondeterminism is not a bug; it's the defining characteristic you must build around.

03 Unreliable Networks: The Six Indistinguishable Scenarios

?

You send a request → receive no response

Six equally plausible explanations, all identical from your perspective:

Scenario Description What You See
① Request Lost Network drops your message before it reaches the remote node No response
② Delayed Arrival Request received but arrives much later than expected No response (yet)
③ Node Crashed Remote node failed; process no longer running No response
④ Paused Node Remote node paused (GC, load spike, VM preemption); will recover No response (yet)
⑤ Response Lost Remote node processed successfully; response lost in network No response (but action succeeded!)
⑥ Response Delayed Remote node processed; response delayed in network queues No response (yet)
CORE PROBLEM Asynchronous packet networks provide zero guarantees on delivery timing or arrival at all. It is impossible to distinguish these scenarios on timeout.

Network Fault Empiricism

Faults per month (medium datacenter) ~12 network faults
Single machine failures ~50%
Entire rack failures ~50%

Real-World Network Fault Examples

  • 🔴 EC2 Transient GlitchesAWS's own infrastructure notorious for causing packet delays and drops
  • 🔴 Software Switch UpgradeTopology reconfiguration caused >1 minute packet delays during upgrade window
  • 🔴 Physical DamageUndersea cables damaged by shark bites affecting intercontinental links
  • 🟠 Asymmetric FaultsNIC (network interface card) drops inbound packets while successfully sending outbound traffic
  • 🟠 Human MisconfigurationTop-of-rack & aggregation switch failures often due to config errors, not hardware defects

💡 Critical Design Lesson

Even rare network faults require explicit handling in your application. Unhandled faults lead to deadlock, data deletion, or arbitrary bad behavior. You cannot assume "rare = won't happen."

04 Detecting Faults & The Timeout Problem

Explicit Fault Signals: Why They Fail

✅ What Works

  • Positive application response — Only reliable proof of success
  • Explicit application-level messages acknowledging completion
  • End-to-end application semantics

❌ What Doesn't

  • TCP RST/FIN (only if OS running; not node crashed)
  • In-band crash notifications (requires scripting; fragile)
  • Network switch management interfaces (unavailable in datacenters)
  • ICMP Destination Unreachable (same limitations as any network packet)

The Timeout Paradox: Theory vs Reality

Fictitious Synchronous System Model

Ideal assumptions:

max network delay = d

max request processing time = r

timeout should be = 2d + r (request + processing + response)

⚠️
Unbounded Delays
Asynchronous networks have NO upper bound on delay. d is infinite; timeout is arbitrary.
⚠️
Nondeterministic Processing
Server implementations cannot guarantee max processing time. One transient spike breaks all timing assumptions.
📊
The Paradox
Long timeout: Long wait to detect failure. Short timeout: Risk of premature declaration → cascading failure.

Network Queueing: The Real Source of Variability

Primary source of packet delay variability is NOT rare packet loss, but packet queueing at multiple layers:

1

Switch Queue

Multiple nodes send to same destination; packets queue one-by-one waiting for egress port

2

OS Input Queue

CPU busy; incoming request queued by kernel, delayed until CPU available

3

VM Pause

Virtual machine paused 10–100ms while hypervisor schedules another VM; incoming data buffered

4

TCP Flow Control

Sender self-limits transmission rate; additional queueing at sender before network entry

5

Cascading Retransmission

TCP timeout waits if packet lost; application sees original delay + retransmit RTT

TCP vs UDP Trade-off

TCP

Reliable delivery but variable delays (retransmits, flow control, packet reordering).

Guaranteed delivery Unpredictable delay

UDP

Variable but avoids some delay sources. Used in VoIP/videoconferencing where delayed data is worthless.

No guaranteed delivery Lower latency
💡
Timeout as a heuristic: Timeouts cannot distinguish scenarios; they're statistical guesses based on observed latencies. Set based on empirical measurements of your specific network, then add safety margin.

05 Byzantine Faults & Fault Models

Byzantine Fault Definition

Node Behavior Spectrum
HonestCorrect behavior
CrashStops; doesn't lie
SlowDelayed response
ByzantineLies; sends corrupt data
  • 🔴 Byzantine FaultNode deliberately sends faulty/corrupted responses; may claim to have received messages it didn't receive
  • 🔵 Byzantine Fault-Tolerant (BFT) SystemContinues operating correctly even if some nodes malfunction, disobey protocol, or are compromised by malicious attackers
  • 🟢 DDIA Assumption: "Unreliable but Honest"Nodes may be slow/unresponsive/outdated, but don't lie if they respond

When BFT is Relevant (3 Cases)

① Aerospace & Safety-Critical

Radiation-induced memory corruption in CPUs/registers → unpredictable node behavior. Examples: aircraft flight control, SpaceX rockets with fault-tolerant computing.

Rare but catastrophic

② Multi-Org P2P Systems

Untrusting parties (Bitcoin, blockchains) need consensus without central authority. Nodes are adversaries; must tolerate deliberate deception.

Consensus-critical

③ NOT typical datacenters

Single-org control + low radiation = cost of BFT protocols prohibitive. Traditional mechanisms (auth, encryption, firewalls) remain primary defense.

Cost-prohibitive

Why BFT is Impractical for Most Systems

🔗
Supermajority Requirement
Need >2/3 nodes functional. 4-node system → max 1 failure tolerated. Expensive redundancy.
🐛
Bug Immunity Illusion
Same software on all nodes → BFT can't protect against bugs. Need 4 independent implementations (cost explosion).
🔐
Security Dependency
If attacker compromises one node, likely compromises all (same software). Firewall & auth remain critical.
⚙️
Complexity
BFT protocols "quite complicated." Hardware support needed for fault-tolerant embedded systems.

The Byzantine Generals Problem

Two Generals Problem: Two armies must agree on battle plan via unreliable messengers (messengers may be delayed/lost). Generalization → Byzantine Generals Problem: n generals must reach consensus; some are traitors sending false messages.

NAMING ETYMOLOGY "Byzantine" chosen to mean "excessively complicated, bureaucratic, devious" (political reference). Avoids offending any specific nationality—pure metaphor for systemic complexity and deception.

06 Safeguards Against Weak Forms of Lying

Without full Byzantine Fault Tolerance, deployable mechanisms guard against non-adversarial faults:

Network Packet Corruption

💻
Problem
Hardware bugs, OS/driver/router bugs; corruption sometimes escapes TCP/UDP checksums (checksums imperfect).
✅
Mitigation: Application-Level Checksums
Simple, pragmatic: add hash/CRC of payload at application layer. Catches errors checksums miss.

Input Validation & Sanitization

Publicly Accessible Apps

  • Strict input validation
  • Range checks on numeric inputs
  • String size limits (prevent DoS allocations)
  • Protocol parsing sanity checks

Internal Services (Behind Firewall)

  • Can relax some checks (less hostile input)
  • BUT basic sanity-checking still needed
  • Protocol parsing must be robust
  • Still assume some inputs malformed

NTP Client Robustness

1

Contact Multiple Servers

NTP client queries multiple time servers simultaneously

2

Estimate Errors

Calculate error bounds on each response; track uncertainty

3

Detect Outliers

Misconfigured/incorrect servers identified as outliers in time range

4

Exclude Outliers

Discard outlier responses; synchronize using consensus time range

💡 Practical Insight

NTP's multi-server approach is more robust than single-server trust. Consensus on time range (rather than exact value) naturally defends against Byzantine-like misbehavior without full BFT overhead.

07 System Models: Timing & Fault Models

Three Timing-Based System Models

Model Characteristics Realism
Synchronous Bounded network delay d, bounded process pauses, bounded clock error. Fixed upper bounds provably exist. ❌ Unrealistic
Unbounded delays & pauses do occur
Partially Synchronous System normally behaves synchronously (bounded delays) but occasionally exceeds bounds. Exceptions are transient. ✅ Most Realistic
Matches real systems
Asynchronous No timing assumptions; no clock; no timeouts possible. Nodes operate without synchronized time. ⚠ Very Restrictive
Rare in practice

Three Node Fault Models

Model Behavior Recovery
Crash-Stop Node suddenly stops, never returns Gone forever
Crash-Recovery Node crashes, may restart after unknown time; stable storage (disk) persists, in-memory state lost Recovers; data survives
Byzantine (Arbitrary) Nodes can do anything: lie, trick, deceive, send corrupted data Requires consensus algorithms

🏆 Practical Recommendation

Partially Synchronous + Crash-Recovery Faults = most useful model for real systems. Assume bounded delays most of the time; design for occasional exceptions. Nodes crash but restart with disk data intact.

How often to assume synchronous behavior 99%+ of time
Exceptions you must handle 1% of operations

08 Correctness: Safety vs Liveness Properties

Example: Fencing Token Algorithm

Token Generation Properties
1. Uniqueness (SAFETY): No two requests return same token value
2. Monotonic Sequence (SAFETY): If request x completes before request y begins, then token_x < token_y
3. Availability (LIVENESS): Node requesting token eventually receives response (if doesn't crash)

Safety: "Nothing Bad Happens"

🔴
Safety Violation = Irreversible Damage
