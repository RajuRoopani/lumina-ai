02 Three Pillars Framework

The Three Dimensions of System Design
Reliability Works correctly under adversity
Scalability Copes with increased load
Maintainability Easy to modify and operate

Every production system must balance these three dimensions. They often trade off against each other—investing heavily in reliability might reduce scalability; optimizing for scale can complicate maintenance.

KEY INSIGHT The chapter focuses primarily on Reliability and Scalability, with Maintainability addressed conceptually throughout.

03 Reliability: Working Correctly Under Adversity

💎 Core Definition

A system is reliable if it performs its desired function correctly, delivers the expected performance even when things go wrong. This requires designing fault-tolerance mechanisms to prevent component faults from cascading into system failures.

🛠️
Fault vs. Failure
Fault: A component deviates from spec. Failure: System stops providing service. Faults are inevitable; failures can be prevented through tolerance design.
🔄
Chaos Engineering
Netflix Chaos Monkey deliberately triggers faults to test fault-tolerance mechanisms. Critical bugs often caused by poor error handling in rare edge cases.
📊
Design Shift
Modern cloud platforms shift from preventing hardware faults to tolerating them via software. Rolling upgrades, machine loss tolerance, and elasticity.

04 Faults & Failures: Definitions & Distinctions

✅ Fault

  • Definition: Component deviates from its spec
  • Scope: Localized to one part of the system
  • Manageable: Can be anticipated and tolerated
  • Examples: Disk failure, process crash, network partition
  • Probability: Impossible to reduce to zero

❌ Failure

  • Definition: System as a whole ceases to provide service
  • Scope: End-user visible impact
  • Preventable: Through fault-tolerance design
  • Examples: Service down, response timeout, data loss
  • Goal: Minimize frequency and recover quickly

The fundamental challenge: Build systems that tolerate faults rather than trying to prevent them entirely. Hardware redundancy, process isolation, retry logic, and graceful degradation are core fault-tolerance strategies.

05 Hardware Faults: Scale, Mitigation & Evolution

1

Mean Time To Failure (MTTF)

Hard disk MTTF is 10–50 years on average. At datacenter scale, hardware faults become guaranteed daily events.

2

Cluster Math: 10,000 Disks

With 10,000 disks and ~10 year MTTF, expect approximately 1 disk failure per day in a large cluster. This isn't a rare edge case—it's normal operational reality.

3

Traditional Mitigation (Hardware Redundancy)

RAID arrays, dual power supplies, hot-swappable CPUs, backup power generators. Single-machine reliability keeps systems up, but complexity increases.

4

Modern Shift: Software Fault-Tolerance

Cloud platforms (AWS, Azure, GCP) prioritize elasticity over single-machine reliability. VMs can fail without warning. Enable rolling upgrades, quick recovery, and multi-node resilience.

06 Software Faults: Harder, Correlated, Systematic

🐛
Dormant Bugs
Software faults are harder to anticipate than hardware. A dormant bug can suddenly trigger across all nodes when an unusual condition occurs (e.g., leap second June 30, 2012 Linux bug).
🔗
Correlated Failures
Unlike hardware faults (random, independent), software faults often affect multiple instances simultaneously, causing system-wide outages.
⚠️
Hidden Assumptions
Root cause is often an unstated assumption. Example: runaway process consuming shared resources (memory, CPU) → cascading failure across the cluster.
🛡️
Mitigation Layers
No silver bullet. Combine testing, process isolation, crash/restart, monitoring, self-checking (e.g., message queue validates incoming = outgoing), and alerts.

Real-World Example: Service Degradation

  • 1️⃣ Cascading Failures One slow backend service causes client timeouts, which retry, which overloads the service further, which causes more timeouts—positive feedback loop
  • 2️⃣ Resource Exhaustion A bug leaks memory or file handles; process eventually crashes or becomes unresponsive
  • 3️⃣ Kill Chain One failure triggers defensive mechanisms (circuit breakers, shedding load) which create secondary failures in dependent systems

08 Scalability: Handling Increased Load

📈 Core Definition

Scalability is a system's ability to cope with increased load. It is not a one-dimensional property you either "have" or "don't have." Instead, scalability is deeply context-dependent: it depends on which load parameters matter most for your service.

🎯
Load is Multidimensional
Requests/sec, read:write ratio, active concurrent users, cache hit rate, working set size. Different services have different bottlenecks.
🔍
Context Matters
A mail server's scalability metric (messages/sec) differs from a social network (fan-out complexity) or a data warehouse (query complexity).
⚠️
Trade-offs Emerge
Optimizing for one load parameter often makes others worse. Twitter's shift from read-time to write-time fan-out shows this tension perfectly.

10 Twitter Timeline: Two Approaches & Hybrid Solution

Challenge: Fan-Out Problem

Each user follows many people; each user is followed by many others. When someone posts a tweet, the system must decide: deliver now (write) or deliver on-demand (read)?

Approach 1: Read-Time Fan-Out

❌ Original Strategy (Deprecated)

Architecture: Post tweet → insert into global tweet table. Home timeline read → join followers/tweets/users on-the-fly.

Problem: 300k reads/sec with expensive multi-table joins overwhelmed the database.

Database bottleneck Slow joins Not scalable

Approach 2: Write-Time Fan-Out

✅ Adopted by Twitter

Architecture: Post tweet → look up all followers in cache, insert tweet into each follower's timeline cache. Home timeline read → already computed (simple, fast read from cache).

Math: 4.6k tweets/sec avg × 75 followers = ~345k writes/sec to cache

Benefit: O(1) home timeline reads. Cost: Heavy write load on cache.

Fast reads Heavy writes Cache memory pressure

The Celebrity Problem & Hybrid Solution

🎭 Celebrity Tweet Scenario

A celebrity with 30M followers posts a tweet. Write-time fan-out would require 30M cache writes in ~5 seconds. This is impractical and causes tail latency for the celebrity's write request itself.

Hybrid Solution: Fan-out most users at write-time (cache pre-compute). For high-follower celebrities, fetch tweets at read-time (merged with user's home timeline on-the-fly). This balances write load and read latency.

11 Performance Metrics: Latency, Response Time & Percentiles

Latency vs. Response Time (Critical Distinction)

⏱️ Response Time

  • Perspective: From the client
  • Includes: Network delay to server + queueing at server + service time + network delay back
  • What users experience
  • Highly variable due to external factors
  • Example: "The API call took 250ms"

🔒 Latency

  • Perspective: From the server
  • Definition: Time a request waits in queue before being handled
  • Excludes: Network delays, actual service time
  • System resource constraint (queue depth, CPU, disk I/O)
  • Example: "Request queued for 50ms before processing"

Why Distribution Matters: Mean vs. Percentiles

CRITICAL METRIC MISUNDERSTANDING Many teams report mean/average response time, but this hides outliers. If 99 requests are fast (50ms) and 1 is slow (5s), mean is only 100ms—but 1% of users had a terrible experience.
1

Percentiles Reveal the Distribution

p50 (Median): 50% of requests faster; half slower. Typical user experience for average case.

2

High Percentiles Show Tail Latency

p95, p99, p99.9: The slowest 5%, 1%, 0.1% of requests. Directly impact user satisfaction—users notice when something is slow.

3

Business Impact of Latency

Amazon: 100ms response time increase → 1% sales reduction. Google: 1-second slowdown → 16% drop in customer satisfaction.

4

SLO/SLA Definition

Example: Median response < 200ms, p99 < 1s, uptime 99.95%. Percentile-based SLOs avoid gaming the metric.

Why Optimize Tail Latency?

👥
User Impact
Slow requests are visible to end-users. Even 0.1% of users (1 in 1000) is a large number at scale. Negative experience spreads via word-of-mouth.
💰
Diminishing Returns
Optimizing p99.9 is expensive (Amazon found it not cost-justified). Focus on p99 or p95 where ROI is high.
🔀
Tail Amplification
Critical finding: If an end-user request requires N backend calls in parallel, the chance of a slow end-user response increases exponentially. One slow backend call makes the entire request slow.

Tail Latency Amplification in Depth

📊 The Math

If each backend call has p99 = 100ms (99% complete within 100ms), and the end-user request requires 4 parallel backend calls:

  • Probability all 4 fast: 0.99^4 ≈ 96%
  • Probability at least 1 slow: 1 - 0.96 = 4%
  • End-user p99 is now ~4%, much worse than individual backend p99

Implication: As services become more interconnected (microservices, APIs), tail latency amplification becomes a dominant factor. Even small improvements to individual p99 values compound at the system level.
