02 Data Representation Fundamentals

The Encoding Journey
In-MemoryObjects, structs, pointers
→ Encode
Byte SequenceNetwork/disk format
→ Decode
In-MemoryObjects reconstructed

The core tension: in-memory representations use pointers and CPU-optimized layouts, but encoded representations must be self-contained, portable byte sequences. The gap between them is where encoding/decoding happens.

WHY THIS MATTERS Your application can't send pointers across a network or persist them to disk. You must serialize objects into a wire format and deserialize them on the other end.

Key Terminology

  • 🔵 Encoding (serialization/marshalling) — Convert in-memory representation to byte sequence for transmission or storage
  • 🟢 Decoding (parsing/deserialization/unmarshalling) — Reconstruct in-memory representation from encoded byte sequence
  • 🔴 Schema — Formal specification of data structure (optional for some formats, mandatory for others)
  • 🟣 Field Identity — How fields are recognized (by name, position, or numeric tag)

03 Compatibility Axes & Deployment Strategy

1

Backward Compatibility

Newer code reads old data. Producer wrote with schema v1; consumer code is v2. New code must understand old wire format. Typically easy: ignore unknown fields, provide defaults for missing fields.

2

Forward Compatibility

Older code reads new data. Producer wrote with schema v2; consumer is still v1. Old code encounters fields it doesn't recognize. Must skip gracefully. Harder to guarantee without explicit design.

3

Rolling Upgrade

Staged deployment strategy. Deploy new version to few nodes at a time. Old and new versions coexist, communicate via shared format. Requires both backward and forward compatibility. Enables zero-downtime deployments.

🏆 Why This Matters for Distributed Systems

You cannot stop the world and deploy everywhere at once. Some nodes will run v1, others v2, temporarily. Your data format must evolve safely or deployments become risky operational events.

Backward Compatibility Difficulty Easy
Forward Compatibility Difficulty Hard
Rolling Upgrade Difficulty Very Hard

04 Language-Specific Encodings: Why NOT

🚨
Critical Flaw
Remote Code Execution (RCE) — Deserializing untrusted data can instantiate arbitrary classes, running attacker code.
🔒
Language Lock-In
Java Serializable, Ruby Marshal, Python pickle only work within their ecosystems. Multi-language services cannot use them.
⚠️
Versioning Afterthought
Schema evolution was bolted on post-hoc. No native support for safe field addition/removal.
📦
Bloated Encoding
Java serialization notorious for large wire size. Include class metadata, field names, type info with every object.
VERDICT Use language-specific encodings only for transient, trusted data (e.g., local caching). Never for network/storage data or untrusted sources.

✅ Pros

  • Zero boilerplate — native language serialization
  • Preserves all object state automatically
  • No schema definition needed

❌ Cons

  • Remote code execution vulnerability
  • Single-language only
  • Hard to evolve safely
  • Often larger than alternatives
  • Tightly couples code to data format

05 JSON, XML, CSV: Human-Readable Trade-Offs

Format Readability Type Safety Verbosity Best Use Case
JSON ✔ High ⚠ Partial (strings vs. numbers) ⚠ Medium Web APIs, logs, config
XML ✔ High ⚠ Optional schema ✘ Very High Legacy enterprise systems
CSV ✔ Medium ✘ None ✔ Low Data export, spreadsheets

Critical Numeric Ambiguities

1

XML/CSV: No Native Types

Cannot distinguish 123 (number) from "123" (string). Parser sees only bytes. Schema required to interpret.

2

JSON: No int/float Distinction

JSON spec distinguishes strings from numbers, but treats all numbers as IEEE 754 double-precision. Cannot represent arbitrary precision integers.

3

Twitter ID Bug

Tweet IDs are 64-bit integers. JavaScript Number type (IEEE 754) loses precision above 2^53. Twitter API returned IDs twice: as JSON number (truncated) and as decimal string (exact). Workaround for client safety.

PRECISION HAZARD Never rely on JSON numbers for IDs, timestamps, or financial data if JavaScript clients consume them. Use strings and parse explicitly.

Binary String Encoding Problem

JSON/XML/CSV have no native support for binary data. Workaround: Base64 encoding. Downside: +33% size overhead.

Binary Data Size Inflation (Base64) +33%

Schema Adoption & Evolution

  • 🔵 JSON Schema — Technically standardized but adoption low; often ignored. Consumers manually code defaults.
  • 🟢 XML Schema (XSD) — Widespread for enterprise; complex to learn/implement
  • 🔴 CSV Schema — No standard; always manual — adds friction to evolution

💡 When to Use Text Formats

JSON/XML excel at cross-organizational data interchange. The standardization value and human-readability often outweigh efficiency costs. Use for public APIs, config files, long-term archives. Do not use for internal high-throughput RPC.

06 Binary Encodings for JSON/XML

Hybrid approach: compress JSON/XML structure into binary while preserving the data model. Examples: MessagePack, BSON, UBJSON, Smile (JSON); Fast Infoset, WBXML (XML).

⚙️
Key Characteristic
Field names encoded in every message (no schema assumed). This bloats the wire format relative to tag-based systems like Protobuf.
🎯
Advantage
Extend JSON/XML with native types (int vs. float, binary strings). Still human-tool-readable with schema knowledge.
📊
Size Improvement
Modest — often 15–25% smaller than text. ROI questionable vs. loss of readability.

MessagePack Case Study: Example 4-1

Data: {"userName": "Martin", "favoriteNumber": 1337, "interests": ["daydreaming", "hacking"]}

VERDICT Binary text formats (MessagePack) only make sense when textual size is a bottleneck AND you need human inspection capability via external tools. Otherwise, move to Protobuf/Thrift for real compression.

07 Thrift & Protocol Buffers: Schema-Driven Architecture

Schema-to-Code Pipeline

→ Codegen
Language Classes(Go, Java, Py)
→ App Code
Encode/Decode(native)

Thrift IDL Example

struct Person {
  1: required string       userName,
  2: optional i64          favoriteNumber,
  3: optional list<string> interests
}

Protocol Buffers Definition

message Person {
    required string user_name       = 1;
    optional int64  favorite_number = 2;
    repeated string interests       = 3;
}

Core Innovation: Field Tags

Instead of encoding field names, both Thrift and Protobuf use numeric tags (1, 2, 3). Tags are the field identity — names are for humans only. This enables safe evolution:

  • 🔵 Add field with new tag → Old code skips unknown tag; new code has default
  • 🟢 Rename field → Tag number unchanged; field name is metadata only
  • 🔴 Remove field → Retire tag; field should not be reused (compatibility issues)

Thrift Binary Encoding

BinaryProtocol (Full type annotations) 59 bytes
CompactProtocol (Optimized) 34 bytes

CompactProtocol achieves ~42% reduction over BinaryProtocol by packing field type + tag into single byte and using variable-length integers.

Thrift vs. Protobuf Comparison

CONSENSUS For internal RPC and storage: Protobuf/Thrift win. Tag-based identity + varint encoding + schema enforcement beat all competitors on safety and efficiency combined.

08 Varint Encoding: The Secret Weapon

Variable-length integer encoding (LEB128) is the compression technique behind Protobuf and Thrift's wire efficiency. Small integers use 1 byte; large ones use more.

Varint Bit Layout
Byte 1:  [Continuation bit (1)] [7 bits of data]
Byte 2:  [Continuation bit (1)] [7 bits of data]
...
Byte N:  [End bit (0)]          [7 bits of data]

Example: 0 = 0x00 (1 byte)
Example: 300 = 0xAC 0x02 (2 bytes)
         = (0x2C | 0x80) (0x02)  [little-endian]

Encoding Examples

Value Bytes Efficiency
0 0x00 ✔ 1 byte
127 0x7F ✔ 1 byte
128 0x80 0x01 ⚠ 2 bytes
300 0xAC 0x02 ⚠ 2 bytes
2^28 - 1 5 bytes ✘ 5 bytes

💡 Why Varint Dominates

Most integers in real systems are small (counts, IDs up to millions, timestamps). Varint exploits this distribution. Typical gain: 50–70% size reduction vs. fixed 32/64-bit encoding for integer-heavy data.

Thrift & Protobuf Type System

  • 🟣 Varint types (int32, int64, uint32, uint64) — Use variable-length encoding
  • 🟣 Fixed types (fixed32, fixed64) — Always 4 or 8 bytes (for non-compressible data)
  • 🟣 Double — IEEE 754 8-byte float
  • 🟣 String / Bytes — Length-delimited (varint length + data)
  • 🟣 Nested messages — Encoded with length-prefix (TLV: Tag-Length-Value)
  • 🟣 Repeated fields — No array type; use repeated scalar (each encoded individually)
  • 🟣 Maps — Represented as repeated key-value pair messages
DESIGN PRINCIPLE Varint + tag-based field identity = automatic backward/forward compatibility. Unknown tags skip; missing fields use defaults. No explicit version numbers needed.

09 Apache Avro: Position-Based Evolution

Avro is a schema-driven system optimized for streaming and data lakes (Kafka, Hadoop). It differs fundamentally from Protobuf/Thrift: no field tags.

Avro Schema Example

{
  "type": "record",
  "name": "Person",
  "fields": [
    {"name": "userName", "type": "string"},
    {"name": "favoriteNumber", "type": ["null", "int"]},
    {"name": "interests", "type": {"type": "array", "items": "string"}}
  ]
}

Writer Schema vs. Reader Schema

Avro's key innovation: two schemas involved in decoding.

1

Writer Schema

The schema used to encode data (embedded in message or registry). Describes what was written.

2

Reader Schema

The schema the consumer code expects. Describes what application wants to read.

3

Schema Resolution

Avro compares writer vs. reader schemas field-by-field and adapts. Can handle renames, reordering, type coercion (int→long).

✅
Advantage: Union Types
Avro naturally supports union types (e.g., ["null", "int"] = optional int). Protobuf requires awkward oneof.
⚠️
Disadvantage: Field Removal
