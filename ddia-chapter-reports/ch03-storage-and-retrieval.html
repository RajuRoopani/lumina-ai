The Core Trade-off

Every database fundamentally performs two operations: store data and retrieve data. The foundational trade-off is unavoidable: well-chosen indexes dramatically accelerate read queries, but every index decelerates write operations. Application developers must understand storage engine internals to select the right engine for their workload.

What is an Index?

An index is a derived data structure that acts as a signpost into the main data. It does not affect the database contents—only query performance. The challenge: indexes must be manually selected for creation; most databases do not automatically index all columns. The developer's responsibility is matching index structures to query patterns.

💡
Index Principle
Index = performance lever. Trade write cost for read speed. Requires conscious design decisions.
⚠️
Manual Selection
Most databases do NOT auto-index all columns. Developers must explicitly decide what to index.
🎯
Knowledge Required
Understanding storage engine internals is essential for optimal database design.

Core Architecture

Hash indexes use a simple append-only log mechanism paired with an in-memory hash table:

1

Write Operation

Append key-value pair to the end of the data file; update the in-memory hash map with the key and the new byte offset of this entry.

2

Read Operation

Look up the key in the hash map to retrieve the byte offset. Seek to that location in the file and read the value.

3

Single Disk Seek

If the value is not in the filesystem cache, reading requires only one disk seek—highly efficient for random access.

Advantages of Append-Only Writes

  • ⚡ Sequential Writes >> Random WritesSequential writes to disk (especially on magnetic drives, but also on SSDs) are dramatically faster than random writes because they avoid seeking.
  • 🔒 Simpler Concurrency & Crash RecoveryAppend-only eliminates the risk of in-place overwrites corrupting data. No locks needed for writes if single-writer thread; immutable segments allow multi-reader.
  • 🧹 No FragmentationData file grows sequentially without the fragmentation patterns that plague file systems with in-place updates.

Fundamental Limitations

🚨
All Keys Must Fit in RAM
The hash map lives entirely in memory. If you have billions of keys, RAM becomes prohibitive. On-disk hash maps suffer from random I/O, poor expansion, and collision handling overhead.
⚠️
No Range Queries
Must look up each key individually. Cannot scan from key A to key Z in sorted order. This is a hard constraint of hash-based indexes.

Bitcask Overview

Bitcask is the default storage engine in Riak. It implements the hash-index pattern with segment management for practical scalability. A critical constraint: all keys must fit in RAM. However, values can exceed RAM—disk seeks retrieve them on demand.

Optimal Workload Match

Bitcask excels when:

Frequent updates per key High Fit
Small number of distinct keys High Fit
High write throughput High Fit
Example: URL → Play Count Perfect

Segment Management & Compaction

As the append-only log grows indefinitely, Bitcask divides it into segments and applies compaction and merging:

1

Threshold-Based Segmentation

When an active segment reaches a size threshold (e.g., 100 MB), it is closed and becomes immutable. A new segment becomes active.

2

Compaction

Remove duplicate keys from closed segments. Retain only the most recent value per key. Discard older entries.

3

Merging

Combine multiple segments into one, further reducing file count and total disk space used.

4

Concurrent Reads/Writes

Background threads handle merging on old segments. Reads and writes continue on the active segment. When merge completes, atomic switch to the merged segment.

Per-Segment Hash Maps

Each segment has its own in-memory hash map. On read:

  1. Check the most recent segment's hash map first
  2. If key not found, check the previous segment
  3. Continue backward to oldest segment

Because merging keeps segment count small (often just 3–5 segments), the lookup overhead remains minimal.

Implementation Considerations

Concern Solution
File format robustness Binary (length-prefixed strings) preferred over CSV for parsing efficiency
Deletion Tombstone marker in log; discarded during merge, not immediate removal
Crash recovery Snapshot hash maps to disk; much faster reload than rescanning log
Partial writes Checksums detect and ignore corrupted log fragments
Concurrent access Single writer thread (enforces sequential log); immutable segments → multiple readers
STRENGTHBitcask is simple, proven, and excellent for workloads with frequent overwrites of a limited key set.

SSTable Definition

A Sorted String Table (SSTable) is a sequence of key-value pairs stored in sorted order by key. Within each merged segment, every key appears only once (duplicates are discarded during merges, retaining the most recent value).

Advantages over Hash-Indexed Logs

1. Efficient Merging

Mergesort-like algorithm. Read from multiple input files simultaneously, compare first keys, copy lowest to output. Scale to thousands of segments.

2. Sparse Index

Don't index every key. One entry per few KB is sufficient. Seek to indexed key, then scan linearly. Dramatically reduce RAM.

3. Block Compression

Group KV pairs into blocks, compress before write. Sparse index points to block start. Saves disk space and I/O.

Sparse Index Example

Instead of indexing every key, maintain just a few strategic entries:

Indexed entries:
  handbag → offset 0
  handsome → offset 50000

To find "handiwork":
  1. Seek to "handbag" (offset 0)
  2. Scan linearly: handbag → handball → ... → handiwork ✓
  3. Stop at "handsome" (offset 50000) — handiwork is sandwiched

Key Insight: Range Queries

✨ Breakthrough Feature

Because SSTables are sorted, you can efficiently scan a range of keys. For example, find all entries from kitty00000 to kitty99999 by seeking to kitty00000 and scanning forward to kitty99999. This is impossible with hash indexes.

Write Path in LSM-Tree

1

Append to Write-Ahead Log (WAL)

Every write is immediately appended to a durability log on disk. This ensures that even if the database crashes, committed writes are not lost.

2

Add to In-Memory Memtable

Simultaneously, add the key-value pair to the in-memory memtable (usually a red-black tree or AVL tree). The tree automatically keeps keys sorted.

3

Continue Writes to New Memtable

When the memtable reaches a threshold (typically a few MB), a background thread flushes it to disk as an SSTable. Concurrent writes continue to a new memtable instance.

Read Path in LSM-Tree

1

Check In-Memory Memtable

First, look up the key in the active memtable. If found, return immediately (fastest path).

2

Check Most Recent On-Disk Segment

If not in memtable, search the most recently flushed SSTable. Use sparse index and binary search or linear scan within a block.

3

Iterate Backward Through Segments

Continue through older segments in reverse order (newest to oldest). Stop at first match or conclude key not found after checking all segments.

Crash Recovery

✅ WAL Durability

  • Memtable loss on crash is handled by WAL
  • Restore memtable from WAL entries at startup
  • Discard WAL log after SSTable flush

⚠️ Recovery Cost

  • Replaying WAL can be slow for large logs
  • Periodic snapshots of memtable reduce replay time
  • Trade-off: storage overhead vs. recovery speed
LSM-TREE DEFINITIONLog-Structured Merge-Tree (LSM-Tree) is an algorithm for merging and compacting sorted files. Originated with Patrick O'Neil et al. (1996). Powers LevelDB, RocksDB, Cassandra, and many modern systems.

The Fundamental Trade-off: RUM Conjecture

🎯 Cannot Optimize All Three

According to the RUM Conjecture (Athanassoulis et al., 2016), no single data structure can simultaneously optimize Read performance, Update (write) performance, AND Memory efficiency. Every system must choose which two to prioritize.

🔵
Hash Index Focus
Read + Update: Excellent on both, but memory efficiency suffers (all keys in RAM).
🩵
LSM-Tree Focus
Update + Memory: Excellent write performance, sparse indexes save RAM, but read amplification fromchecking multiple levels.
🟣
B-Tree Focus
Read + Memory: Balanced I/O, good cache locality, but write amplification from node splits and rewrites.

Modern System Implementations

  • 🔵 LevelDBGoogle's embedded LSM-Tree database; widely used as reference implementation and in Chrome IndexedDB.
  • 🟢 RocksDBFacebook's fork of LevelDB; optimized for high concurrency and SSD performance; powers numerous production systems.
  • 🟠 CassandraDistributed database using LSM-Tree design at scale with log-structured merging across nodes.
  • 🟣 MySQL/InnoDBTraditional B-Tree index structure; balanced design for read-heavy OLTP workloads.

Compaction Strategies

Different LSM implementations use different compaction policies to balance read and write amplification:

Size-Tiered Compaction (Cassandra)

When L0 reaches N SSTables, merge all into one L1 SSTable. Continues up the hierarchy. Trade-off: fewer compactions but larger merges.

Leveled Compaction (LevelDB, RocksDB)

Each level has a fixed target size (e.g., L0=4MB, L1=40MB, L2=400MB). Merges are triggered to maintain level invariants. Trade-off: more frequent but smaller compactions, better read performance.

SSD-Specific Considerations

Modern storage engines must account for SSD characteristics:

Sequential Write Advantage Less Pronounced on SSD
Random I/O Cost Much Lower than HDD
Write Amplification Still Critical
Erasure Block Lifetime Constrained Resource

Write-Ahead Logging (WAL) Variants

Write-Ahead Log (WAL) Durability guarantee; replay on crash Every write = 2 disk ops (log + memtable); slows sequential write throughput

Bloom Filters in LSM Compaction

Bloom filters are probabilistic data structures used to answer "Is this key in this SSTable?" with certainty for yes and probabilistic for no.

OPTIMIZATIONDuring LSM read, check the Bloom filter first. If it says "definitely not present," skip the SSTable entirely. Eliminates many disk seeks, dramatically reducing read amplification.

Covering Indexes & Full-Text Search

Covering indexes store the actual value within the index, not just the key and offset. This enables index-only scans without accessing the main data file.

Full-text search (Lucene-style inverted indexes) builds separate indexes per word, mapping word → list of document IDs. Merges use the same segment-based approach as LSM-Trees.

🟢
Bloom Filter Benefit
Space/Time trade-off: 1% false positive rate saves orders of magnitude in disk seeks.
🔵
Fuzzy Matching
Levenshtein automata enable typo-tolerant search at scale; 100× faster in modern Lucene versions.

09 RUM Conjecture: The Fundamental Design Constraint

RUM Triangle: Pick Two Corners
Read      Update      Memory

No single index structure can simultaneously optimize all three

The RUM Conjecture Framework

Athanassoulis et al. (2016) in "Designing Access Methods: The RUM Conjecture" established that any data structure optimizing storage and retrieval must balance three competing objectives:

Read Efficiency

Cost of accessing stored data. Lower is better. Achieved via indexes, caching, spatial locality.

Update Efficiency

Cost of writing/modifying data. Lower is better. Append-only and sequential writes minimize this.

Memory Efficiency

Space overhead of index structures. Lower is better. Sparse indexes and compression help.

Implications for System Design

1

Understand Your Workload

Is it read-heavy (OLTP), write-heavy (streaming, time-series), or mixed? This determines which corner of the RUM triangle to prioritize.

2

Accept Trade-offs

No single engine excels at all three. Acknowledge the cost: hash indexes waste RAM, LSM reads hit multiple levels, B-Trees rewrite pages.

3

Mitigate Weak Dimensions

Use secondary structures (caching, Bloom filters, covering indexes) to address the dimension you're not optimizing for.

4

Tune Aggressively

Modern systems (RocksDB, Cassandra) expose knobs for tuning compaction strategy, block size, cache size, and more. Invest time in profiling.

Index Fundamentals Recap

Indexes are essential for query optimization but impose write-time costs. The decision to index is a balance—add indexes for frequently queried columns, avoid redundant indexes, and monitor query patterns.

Covering Indexes (Index-Only Scans)

A covering index stores the actual column values in the index structure, not just the key and pointer to the main table. This allows a query to be satisfied entirely from the index without accessing the main data file.

Example: Index on (user_id, email, age)
  User 123 → email "alice@example.com", age 28
  
Query: SELECT email, age WHERE user_id = 123
  Result: Served directly from index (no main table access)

Full-Text Search (Inverted Indexes)

Full-text search engines (Lucene, Elasticsearch) build inverted indexes mapping each word to the list of documents containing it. Construction mirrors LSM-Tree segment merging:

  • Index documents in-memory as they arrive
  • Periodically flush to disk as a segment
  • Merge segments in the background using similar algorithms as LSM
  • Supports fuzzy/approximate matching via Levenshtein automata (100× faster in modern versions)

Spatial Indexing (Multi-Dimensional)

For multi-dimensional data (geographic coordinates, image feature vectors), spatial indexes like the UB-Tree (Ramsak et al., 2000) map n-dimensional points to a 1-dimensional linear order while preserving spatial locality. Enables range queries across multiple dimensions.

Index Selection Strategy

✅
Index This
Frequently queried columns, especially in WHERE clauses, JOIN conditions, and ORDER BY operations. High cardinality. Selective filters.
❌
Avoid Indexing
Low cardinality (gender, status). Columns rarely queried. Tables with very frequent writes and infrequent reads. Redundant indexes (duplicate of existing compound index).
⚠️
Profile & Monitor
Use query logs and explain plans. Identify slow queries. Test index impact with production-like data volumes. Remove unused indexes periodically.

Composite & Partial Indexes

Composite indexes (multi-column) are effective for queries filtering on multiple columns or using a prefix scan. Order matters: put the most selective column first.

Partial indexes index only rows satisfying a WHERE condition (e.g., active users only). Reduces index size and write overhead for cold/inactive data.

Bloom Filters: Answering "Does This Key Exist?"

A Bloom filter is a space-efficient probabilistic data structure (Burton H. Bloom, 1970). It answers membership queries with:

  • ✔ Definite "no": key is certainly not in the set
  • ⚠ Probabilistic "yes": key might be in the set (false positive rate ≈ 1%)

How Bloom Filters Work

1

Hash Multiple Times

Hash the key using k independent hash functions, each producing an offset into a bit array.

2

Set Bits During Insert

Set the bits at all k positions to 1 when inserting a key. Multiple keys may set overlapping bits.

3

Test on Lookup

Check if all k bits are set. If any bit is 0, key is definitely absent. If all are 1, key might be present (false positives possible).

4

LSM Application

Store a Bloom filter in each SSTable. On read, check filter before seeking disk. Eliminates many fruitless disk accesses.

Space/Time Trade-off

EFFICIENCY GAINA 1% false positive rate requires ~10 bits per element. To read 1 billion keys across 100 SSTables (100 segments checked = 100 disk seeks worst case), Bloom filters reduce this to ~1 seek on average by eliminating 99% of false positives.

Burst Tries for String Keys

Burst Tries (Heinz et al., 2002) are a specialized index structure for string-based keys. They perform faster than traditional hash tries by bursting into sorted arrays when a prefix bucket reaches a size threshold. Excellent for autocomplete and prefix-matching queries.

Other Probabilistic Approaches

  • 🔵 Cuckoo HashingVariant of hash tables with guaranteed O(1) lookups and higher space efficiency than standard hash tables.
  • 🟢 Count-Min SketchApproximate frequency counter; answers "how many times did this event occur?" with bounded error.
  • 🟣 HyperLogLogApproximate cardinality estimator; answers "how many unique items?" with < 2% error using constant memory.
  • 🟠 T-DigestStreaming percentile approximation; enables quantile queries on unbounded data streams.
