04 Hash Indexes & Log-Structured Approach

1

In-Memory Hash Map

Maintain a dictionary: key → byte offset in data file. This enables O(1) lookup of where a key is stored on disk.

2

Append-Only Writes

All writes go to the end of an immutable log file. Update the in-memory hash map to point to the new offset. Sequential I/O is orders of magnitude faster than random I/O.

3

Segment Compaction

When the log grows too large, break it into segments and run a background compaction process: discard duplicate keys, keep only the most recent value per key.

4

Merge Segments

Combine multiple compacted segments while maintaining sorted order. Old segments remain readable; switch to new merged segment atomically.

Bitcask Pattern (Riak's Storage Engine)

🎯 Design Goal: Maximize Write Throughput

Bitcask is optimized for workloads where keys fit in RAM but values are large. All keys must reside in-memory hash map; values retrieved from disk with a single seek.

Write Operation Cost 1 append + 1 hash update
Read Operation Cost 1 hash lookup + 1 disk seek
Range Query Cost Individual key lookups (no sorting)

Hash Index Advantages

  • 🟢 Appending is Efficient File appends are "generally very efficient" due to sequential disk access; much faster than random writes
  • 🟢 Concurrency Simplified Immutable segment files enable parallel reads without locks; single writer thread
  • 🟢 Crash Recovery Easy Immutable log enables quick recovery; checksums detect corruption
  • 🟢 No Fragmentation Append-only avoids file fragmentation that plagues in-place updates

Hash Index Limitations

  • 🔴 Hash Table RAM Bottleneck The entire hash map must fit in memory; no way to efficiently page to disk
  • 🔴 No Range Queries Query for kitty00000 to kitty99999 requires individual hash lookups per key—no index ordering
  • 🔴 Linear Scan Without Index Reading all data is O(n) sequential file scan if you have no index

Segment Compaction & Merging Details

Issue Solution in Bitcask
File Format Binary (length prefix + raw bytes), not CSV or JSON
Deletion Append tombstone record; during merging, skip all older versions of that key
Crash Recovery Store snapshots of segment hash maps on disk for fast reload on restart
Partial Writes Checksums detect corrupted segments; discard and replay from before corruption
Concurrency Single writer; segments immutable so reads fully parallel
💡
Bitcask model works best for: Key-value stores where all keys fit in RAM and write throughput matters more than range query speed. Good for time-series data, event logs, caches.

Why SSTables Beat Hash Indexes

⚡
Efficient Merging
Merge-sort algorithm scales to files larger than RAM. Compare smallest key in each input file, copy winner to output, repeat.
💾
Sparse Index
Not all keys indexed. One entry per few KB suffices. Use binary search to locate key range, then linear scan a small block.
📦
Block Compression
Group records into blocks, compress entire block before disk write. Saves space and bandwidth. Index points to block boundaries.
🔍
Range Queries
Because data is sorted, finding kitty00000 to kitty99999 is a contiguous scan—no random seeks required.

LSM-Tree Write Flow (Memtable Based)

1

Append to Write-Ahead Log (WAL)

Unsorted writes go to disk-based log for crash recovery. This log is append-only and can be discarded after the step below.

2

Add to In-Memory Tree

Insert key-value into memtable (red-black tree or AVL tree, kept sorted). This is in-RAM and fast.

3

Memtable Exceeds Threshold

When memtable reaches size limit (e.g., few MB), flush to disk as an immutable SSTable file.

4

Create New Memtable

Start fresh in-memory tree for new writes. Old memtable's SSTable becomes a read-only segment.

5

Background Compaction

Periodically merge and compact segments using size-tiered or leveled strategies. Reduces number of segments, speeds future reads.

LSM Read Flow

  1. Check memtable first (in-memory, fastest)
  2. Check most recent on-disk SSTable if not found
  3. Check progressively older SSTables in reverse chronological order
  4. Return value or "not found" if exhausted all segments

LSM-Tree Definition

Log-Structured Merge-Tree is an algorithm for merging and compacting sorted files, invented by O'Neil et al. (1996, Acta Informatica).

KEY ADVANTAGE LSM transforms random writes into sequential disk writes, yielding higher throughput than B-trees—at the cost of read latency and space amplification.

LSM Performance Optimizations

Technique Purpose Implementations
Bloom Filters Probabilistic set membership test; eliminate disk reads for non-existent keys LevelDB, RocksDB, Cassandra
Size-Tiered Compaction Merge smaller newer SSTables into progressively larger older ones HBase, Cassandra
Leveled Compaction Split key space into fixed-size SSTables per level; older data in separate levels; incremental merging LevelDB, RocksDB

✅ Pros of LSM-Trees

  • Higher write throughput (sequential I/O)
  • Scales to large data sets (compaction works for huge files)
  • Range queries supported (sorted structure)
  • Compression friendly (block-based)

❌ Cons of LSM-Trees

  • Read latency higher (multiple segment checks)
  • Space amplification (compaction overhead)
  • Compaction CPU cost (background threads)
  • Tuning complexity (size-tiered vs leveled)

06 B-Trees: The Update-in-Place Alternative

B-Tree vs LSM-Tree Design Philosophy
B-TreeUpdate-in-Place

Overwrite fixed-size pages on disk. Keeps data sorted. Widely used (InnoDB, PostgreSQL).

LSM-TreeAppend-Only

Append-only log with periodic merging. Sequential I/O efficient. Newer approach (LevelDB, RocksDB).

B-Tree Trade-offs

⚡
Fast Point Reads
Binary search directly to leaf page, single I/O to retrieve key. O(log n) page lookups.
⚠️
Write Penalty: Double-Write
InnoDB double-write buffer: first write to buffer, then to page. Guards against torn writes.
⚠️
Full-Page Writes
PostgreSQL full-page write: log entire page on first modification after checkpoint (Vondra, 2016).
📚
Wide Industry Adoption
Bayer & McCreight (1970). Comer's survey (1979) deemed it "The Ubiquitous B-Tree." Still dominant.

Covering Indexes & Read Optimization

B-tree systems leverage covering indexes (Webb, 2008): if index contains all columns needed for a query, the database avoids the main table read entirely.

SSD CONSIDERATIONS I/O patterns matter significantly on SSDs (Grand, 2014). Random vs. sequential access has less dramatic cost difference than on magnetic drives, but still present.

08 Key Components & Their Roles

  • 🔵 Hash Index (In-Memory) Maps key → byte offset in append-only log. O(1) lookup, but hash table must fit in RAM. Used by Bitcask (Riak).
  • 🟢 Memtable (Sorted Tree) In-memory red-black or AVL tree holding recent writes. When full, flushed to disk as SSTable. Enables batched writes to disk.
  • 🟠 SSTable (Sorted String Table) Immutable sorted key-value file on disk. Supports efficient merging and range queries. Foundation of LSM-tree compaction.
  • 🟣 Write-Ahead Log (WAL) Append-only disk log of all writes before they enter memtable. Enables recovery on crash; can be discarded after SSTable is flushed.
  • 🔴 Bloom Filter Probabilistic data structure (Burton Bloom, 1970). Tests set membership with allowed false positives but no false negatives. Speeds LSM reads by eliminating segments.
  • 🟦 Compaction Process Background thread merging multiple SSTables using merge-sort. Discards old values for duplicate keys, reduces segment count, frees disk space.
  • 🟦 B-Tree Node Fixed-size page (typically 4 KB) holding sorted keys and child pointers. Maintains logarithmic height regardless of dataset size.
  • 🔵 Sparse Index LSM optimization: one index entry per few KB of data. Uses binary search to find key range, then linear scan within block. Reduces index size.
  • 🟢 Block Compression Groups SSTable records into blocks, compresses before disk write. Sparse index points to block boundaries. Saves I/O bandwidth and disk space.
  • 🟠 Crash Recovery Snapshot Disk snapshot of segment hash maps (Bitcask). Enables fast reload of in-memory state on restart without replaying entire log.

Full-Text Search Indexing (Lucene Approach)

Beyond key-value stores, full-text search engines like Lucene use inverted indexes with segment-based architecture (McCandless, 2011, 2013):

  • Inverted Index: Maps term → list of documents containing that term
  • Segment Merging: Visualized in McCandless blog (Feb 2011); similar to LSM-tree compaction
  • Fuzzy Matching: Levenshtein automata (Schulz & Mihov, 2002) for typo-tolerant search
  • FuzzyQuery Optimization: 100x faster in Lucene 4.0 (McCandless, March 2011)
  • String Data Structures: Burst Tries (Heinz et al., 2002) for efficient key handling
🎯
Strategy Selection Rule: If your workload is write-heavy with large keys or values → LSM-tree. If reads dominate and you need predictable latency → B-tree. For full-text → inverted index with segment merging.

10 Trade-offs & Risk Analysis

The RUM Conjecture: Fundamental Limit

Athanassoulis et al. (2016, EDBT) proved that you cannot simultaneously optimize all three:

R
Read Cost
U
Update Cost
M
Memory Cost

Each storage engine makes explicit trade-offs. You must pick which dimension to sacrifice. This is not a tuning problem—it's a fundamental mathematical constraint.

Critical Risks & Mitigations

🚨
Hash Index RAM Limit
All keys must fit in memory (Bitcask). If you have 10M keys, need 10M hash entries. No eviction possible. Mitigation: Use SSTable/LSM for larger datasets.
🚨
LSM Write Amplification
Compaction causes data to be rewritten multiple times (size-tiered more than leveled). With 10GB dataset, may write 100GB total during compaction. Mitigation: Choose leveled compaction; tune thresholds.
⚠️
B-Tree Double-Write Overhead
InnoDB writes each page twice (double-write buffer + main file) to guard against torn writes. Reduces write throughput. Mitigation: Use power-fail safe hardware; tune innodb_doublewrite.
⚠️
Range Query Performance Gap
Hash-indexed logs (Bitcask) cannot efficiently query kitty00000 to kitty99999. Must do individual lookups. Mitigation: Use SSTable/B-tree if range queries are critical.
⚠️
Compaction CPU/Disk Load Spikes
Background compaction threads can steal CPU and I/O from user queries. P99 latency degrades during compaction. Mitigation: Separate compaction thread pools; rate-limit compaction.
💡
SSD vs HDD Considerations
Magnetic drives: sequential >> random (100x+ faster). SSDs: smaller gap but still present. Random I/O patterns matter on both. Tune based on hardware (Grand, 2014).

11 Real-World Implementation Details

File Format Choices

⚙️ Binary vs Text Encoding

Best Practice: Use binary format (length prefix + raw bytes), not CSV or JSON. Binary is faster to parse, smaller on disk, and avoids escaping issues. All production LSM engines use binary.

Deletion & Tombstones

To delete a key in append-only logs, you cannot erase the original value. Instead:

1

Append Tombstone Record

Write a special record marking the key as deleted (e.g., key with null/empty value).

2

During Compaction

When merging segments and encountering the same key in multiple files, keep only the value from the most recent segment. If that value is a tombstone, discard all older versions and skip the key in output.

3

Read Returns Nil

After compaction, key is effectively deleted. Subsequent reads return "not found" or null.

Checksums & Corruption Detection

Protect against partial writes:

  • Per-Record Checksum: Compute CRC32 or SHA1 of each record; store with record
  • Detect Corruption: On read or recovery, recalculate checksum and compare
  • Discard Bad Segments: If checksum fails, that segment is corrupted; discard and replay WAL from before corruption

Concurrency Model

Aspect LSM-Trees B-Trees
Write Concurrency Single writer thread (append to memtable + WAL), ordered writes Locking on modified pages; conflict potential
Read Concurrency Fully parallel (segments immutable); no locks needed Page-level locks; readers block during page modifications
Compaction Impact Background thread; old segments readable during merge Rebalancing happens in-place; can block reads temporarily

Crash Recovery: WAL & Snapshots

  • 🔵 Write-Ahead Log (WAL) All writes appended to disk log before modifying memtable. On crash, replay WAL to reconstruct memtable state.
  • 🟢 Segment Snapshots Periodically save hash map (Bitcask) or compact data structure state to disk. On restart, load snapshot + replay WAL since snapshot = fast recovery.
  • 🟠 ARIES Protocol Mohan & Levine (1992, SIGMOD) formalized recovery with redo/undo logs. Used by many commercial databases for comprehensive crash safety.

Distributed Considerations

12 Performance Patterns & Empirical Analysis

Write Throughput: Sequential vs Random

Magnetic Drive: Sequential Write ~100 MB/s
Magnetic Drive: Random Write ~1 MB/s
SSD: Sequential Write ~500 MB/s
SSD: Random Write ~200 MB/s
INSIGHT On magnetic drives, sequential writes are 100x faster. On SSDs, gap is smaller (~2.5x) but still significant. LSM-trees exploit this by turning random writes into sequential writes.

LSM vs B-Tree: Callaghan's Analysis (Code Mesh 2016)

Jeff Callaghan compared LSM (RocksDB) vs B-Tree performance empirically:

  • LSM Advantage: Higher write throughput on high-load workloads (better utilization of disk I/O bandwidth)
  • B-Tree Advantage: More predictable read latency; lower P99 on point lookups
  • Trade-off: LSM reads suffer higher latency due to checking multiple segments; B-tree reads benefit from single binary search

Tuning RocksDB (Canadi, Dong, Callaghan 2016)

RocksDB offers extensive tuning knobs for balancing read/write/space:

Write-Optimized Tuning

Large memtable (e.g., 128 MB), fewer compaction threads, rate-limit compaction to prevent stalls.

✔ Higher write throughput ✘ Slower reads

Read-Optimized Tuning

Smaller memtable, more aggressive compaction, leveled strategy, block cache tuning.

✔ Faster reads ⚠ Compaction overhead

Space-Optimized Tuning

Compression enabled, leveled compaction, smaller block size, aggressive cleanup.

✔ Lower disk usage ⚠ CPU cost

OLTP vs OLAP Architecture Mismatch

Stonebraker et al. (2007, VLDB) argued that traditional OLTP databases are poorly suited for analytical queries. Harizopoulos et al. (2008, SIGMOD) analyzed this in "OLTP Through the Looking Glass":

❌
OLTP Drawback for Analytics
Optimized for point lookups + small transactions. Analytical queries scan large ranges, aggregate columns → slow on B-tree.
✅
In-Memory OLTP Solution
VoltDB (2014) stores data in RAM, eliminates disk overhead. Good for high-throughput OLTP; separate analytical engine for large scans.
💡
Anti-Caching Technique
DeBrabant et al. (2013) proposed evicting cold data to disk, keeping hot working set in RAM. Hybrid approach between in-memory and disk-based OLTP.

13 Key Takeaways & Critical Insights

🎯
Index Selection is Workload-Dependent: Indexes accelerate reads but slow writes. Choice of index structure (hash, SSTable, B-tree) must match your access patterns. There is no one-size-fits-all solution.
⚡
Sequential I/O Dominates on Magnetic Drives: Appending to a file is ~100x faster than random page overwrites on HDDs. LSM-trees exploit this principle; B-trees suffer from random I/O despite better conceptual simplicity.
🔐
RUM Conjecture is Fundamental: You cannot simultaneously optimize Read cost, Update cost, and Memory cost. Every storage engine sacrifices one dimension. Understand your system's bottleneck and tune accordingly.
🏗️
LSM-Trees Win on Write Throughput: By batching writes in memtable and deferring sorting/merging to background, LSM achieves higher sustained write throughput than B-trees. Cost: higher read latency (check multiple segments) + space amplification (compaction rewrites data).
🔍
B-Trees Provide Predictable Point Reads: Direct binary search to leaf page; no segment checking. Favored for OLTP where latency consistency matters. But random I/O on writes makes them slower for write-heavy workloads.
