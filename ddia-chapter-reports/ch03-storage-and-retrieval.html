Every database engine must solve two problems:

  • 💾 Write PathAccept and durably persist data to disk efficiently; maintain data integrity even after crashes
  • 🔎 Read PathRetrieve requested data with minimal disk I/O; support exact lookups, range queries, and sorted scans

The Index Trade-Off

📇
What Indexes Do
Well-chosen indexes dramatically speed up reads by maintaining auxiliary data structures (hash maps, trees) that map keys to locations on disk.
⚖️
The Write Cost
Every index must be updated when data changes. More indexes = slower writes. Must be explicitly chosen based on query patterns.
🎯
Developer Decision
Developers manually select which indexes to create. This choice is central to storage engine design and performance tuning.
Aspect Without Index With Index
Read Latency ✘ O(n) full scan ✔ O(log n) lookup
Write Latency ✔ Update one location ✘ Update data + index
Disk Space ✔ Minimal overhead ✘ Extra storage for index
Range Queries ✘ Inefficient ✔ Fast sorted iteration

Write Path (Durability-First)

1

Write Request Arrives

Client sends key-value pair to storage engine. Engine must guarantee persistence even if power fails immediately.

2

Append to Log (WAL)

Write-Ahead Log: append write to immutable log file on disk before modifying in-memory structures. Log acts as recovery record.

3

Update In-Memory State

Add to memtable (sorted tree) or hash map in RAM. If crash occurs now, WAL can replay writes on restart.

4

Return ACK to Client

Write confirmed when both log and in-memory updates are durable. Client can proceed with confidence.

Read Path (Lookup Hierarchy)

1

Query Arrives

Client requests value for a key. Engine searches in priority order: memtable first (most recent writes).

2

Check In-Memory Memtable

If key found in RAM memtable (tree structure), return immediately. This hit bypasses disk entirely.

3

Search Disk Segments (Newest First)

If not in memtable, search most-recent SSTable on disk using sparse index. Each SSTable is sorted; binary search finds range.

4

Use Bloom Filter (Optional)

Before disk seek, check bloom filter: space-efficient set membership test. If filter says "not here," skip this SSTable entirely.

5

Return Result

Once found in any segment, return value. If not found anywhere, return "key not found" after checking all segments.

04 Log-Structured Storage Engines

The Bitcask Model: Simplest Append-Only Log

Bitcask Architecture
In-Memory Hash Map:
"user:42" → 145
"user:98" → 8472
"user:150" → 12901

Disk File (append-only):
[offset 0] user:42 = "Alice"
[offset 145] user:98 = "Bob"
[offset 8472] user:150 = "Charlie"
[offset 12901] user:42 = "Alice_Updated" ← duplicate key

Hash map stores byte offsets. Values fetched from disk via single seek. All keys must fit in RAM.

  • 🔵 ArchitectureImmutable data file + in-memory hash map indexing byte offsets; reads require one disk seek, writes are sequential
  • 🟢 AdvantagesSequential writes >> random writes (10–100× faster on spinning disks); simple crash recovery; immutable data safe for concurrent reads
  • 🔴 ConstraintAll keys must fit in RAM memory; values can be large (loaded via single seek), but keyspace is bounded by available RAM
  • 🟠 Use CaseHigh-frequency updates to same small set of keys (e.g., video play counts: one key per video, updated millions of times)
  • 🟣 LimitationNo range queries; cannot efficiently scan keys in range [key1, key2). Must look up each key individually.

Disk Space Management: Compaction & Merging

Problem Appending forever means disk fills with stale data. After millions of writes, same key exists at multiple offsets; only latest matters.
1

Break Log into Segments

Divide append-only log into fixed-size files (e.g., 100 MB each). Newest segment is still writable; older segments are immutable.

2

Compact Each Segment

For each segment, scan from start to end. Keep only the latest value for each key. Discard all earlier versions.

3

Merge Compacted Segments

Combine multiple compacted segments into one (mergesort-like). Even if combined size > available RAM, streaming merge works.

4

Background Process

Compaction and merging happen in background. Old segments remain accessible for reads until merge completes. Then atomically switch to merged file.

✅ Compaction Wins

  • Reduces disk space: removes stale versions
  • Works on unbounded datasets (>> RAM)
  • Disk I/O remains sequential, not random
  • Crash recovery simple: replay log + use hash map

❌ Compaction Costs

  • Disk I/O spike during merge (reads + writes)
  • Cannot efficiently support range queries
  • Disk space usage still high during merge (need space for both old + new files)
  • Keyspace capped by available RAM

🏆 When Bitcask Shines

Riak key-value store uses Bitcask for workloads with small keyspace, many updates to same keys (e.g., user session data, metrics). Write throughput can exceed 100k ops/sec.

Sparse In-Memory Index

SSTable's sorted nature allows sparse indexing—do not store offset for every key:

SSTable Sorted Contents:
"handbag" (offset 0), "handle" (offset 47), "handiwork" (offset 102), "handsome" (offset 205), ...

Sparse Index (sample every 10 KB):
"handbag" → 0
"handsome" → 205

Lookup "handiwork":
1. Index says "between offset 0 and 205"
2. Seek to offset 0, scan forward
3. Find "handiwork" at offset 102
Result: One disk seek + small scan, not random access per key

Block-Level Compression

📦
Grouping
Divide SSTable into logical blocks (e.g., 4 KB). Each block compressed independently (gzip, LZ4, etc.).
💾
Space Savings
Sorted data compresses well (repeated prefixes, similar values). 50–90% space reduction common.
⚡
I/O Benefit
Sparse index points to block starts. Fewer bytes to read from disk = lower I/O cost and network bandwidth.

Efficient Segment Merging (Mergesort-Like)

The killer feature: merge multiple sorted SSTables without loading entire files into RAM:

1

Open Multiple Input Files

Read pointers in each sorted SSTable. Keep one entry buffered from each file.

2

Find Minimum Key

At each step, compare buffered keys. Pick the smallest one.

3

Handle Duplicates

If same key appears in multiple SSTables, keep value from newest segment, discard older versions.

4

Write to Output

Write entry to output file, advance pointer in source file, refill buffer.

5

Complete Merge

Output is fully sorted SSTable. Works even if input files combined size > available RAM (streaming I/O).

Why This Matters Hash-indexed logs cannot efficiently merge because unsorted segments require explicit sorting pass. SSTables leverage sorted order to merge in one streaming pass with minimal overhead.

Component Breakdown

  • 📝 Write-Ahead Log (WAL)Append-only log file on disk. Every write goes here first. Survives crashes; replayed to restore memtable on recovery.
  • 🌳 MemtableIn-memory sorted data structure (red-black tree, AVL tree, or B+ tree). Newest data lives here. Reads check memtable first.
  • 📂 SSTable SegmentsWhen memtable threshold reached (~few MB), flush to disk as sorted SSTable file. Multiple SSTables exist simultaneously on disk.
  • 🔧 Compaction ProcessBackground thread periodically merges SSTables. Removes duplicate keys, maintains sorted order, reduces file count.
  • ⚙️ Bloom FiltersOptional per-SSTable data structure: answer "key exists?" without disk seek. Space-efficient; allows false positives but not false negatives.

Read Path (Multi-Level Search)

1

Check Memtable First

Search in-memory sorted tree. If key found, return immediately (no disk I/O). Hit rate depends on workload.

2

Search Disk SSTables (Most Recent First)

If not in memtable, check most recently written SSTable on disk (highest sequence number). Use sparse index + bloom filter.

3

Bloom Filter Check

Before seeking to disk, check bloom filter. If filter says "not here," skip this SSTable. Saves most disk seeks for missing keys.

4

Iterate Older Segments

If not found in newest segment, search progressively older segments. Stop when key found (most recent value wins).

5

Not Found

If checked all segments and key never appeared, return "key not found." Bloom filter makes this fast (no disk seeks).

Compaction Strategies

⭐ Common

Size-Tiered Compaction

Group SSTables by size. When N SSTables of similar size accumulate, merge all N into one larger SSTable. Simpler, less I/O overhead.

Simple to implement Low write amplification Larger SSTables for old data

Leveled Compaction

LevelDB, RocksDB Organize SSTables into "levels" by age. Level 0 (newest) small. Level 1, 2, ... progressively larger. Incremental merges.

Better read latency More I/O during compaction Higher write amplification
