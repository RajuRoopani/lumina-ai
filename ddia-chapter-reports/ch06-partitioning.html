Partitioning (also called sharding) is the technique of breaking data into partitions and distributing them across multiple nodes in a shared-nothing cluster.

Core Properties

  • 🔴 One Record = One PartitionEach row/document/object belongs to exactly one partition — no duplication, no split ownership
  • 🔵 Independent Small DatabaseEach partition behaves like a miniature database with its own schema, indexes, and storage
  • 🟢 Distributed StoragePartitions span multiple physical machines, disks, and processors — each handles subset of total data
  • 🟠 Synonym ChaosDifferent systems use different terminology: Shard (MongoDB, Elasticsearch), Region (HBase), Tablet (Bigtable), vnode (Cassandra), vBucket (Couchbase)

Fundamental Goals

1. Data Scalability Critical
2. Query Load Distribution Critical
3. Independent Execution High
4. Linear Throughput Scaling Ideal
THROUGHPUT SCALING FORMULA N nodes with no replication = ~N × single-node throughput. With replication factor R, effective scaling = N / R. Example: 10 nodes + 3-node replication = ~3.3× throughput.

A partitioned system splits responsibility across layers:

Layer 1: Partition Assignment

Each partition owns a range of keys (or hash bucket). Assignment can be static or dynamic based on rebalancing.

Layer 2: Request Routing

Client or middleware determines which partition to contact based on the query key. Three common approaches:

  • Smart Client: Client library knows partition map, routes directly
  • Routing Tier: Separate cluster layer forwards requests (HBase, Cassandra)
  • First-Contact: Send to any node; it forwards to correct partition (older systems)

Layer 3: Execution

Each partition processes independently. Single-partition queries require only local disk/memory. Multi-partition queries scatter across multiple nodes and gather results.

🎯 Architecture Insight: Shared-Nothing

No shared disk, no shared memory. Each partition owns its data exclusively. This eliminates contention and allows horizontal scaling. Downside: cross-partition transactions are complex.

04 Key-Range Partitioning

Encyclopedia Model: Contiguous Key Ranges
Vol. 1 (A–B) | Vol. 2 (C–D) | … | Vol. 12 (T–Z) | Vol. 13 (*)

Continuous ranges of key values assigned to partitions

Assign a continuous range of keys to each partition. Example: encyclopedia volumes where partition 1 holds A–B, partition 2 holds C–D, and so on.

How It Works

1

Define Partition Boundaries

Decide minimum and maximum key for each partition. Not necessarily evenly spaced — boundaries chosen to distribute load.

2

Direct Key Lookup

Given a key, find which range it falls into and contact that partition directly. O(log N) with sorted boundaries.

3

Range Scan Efficiency

Keys are sorted within each partition. Scan all records in range [K1, K2] by reading one partition sequentially.

Advantages

✅ Pros

  • Efficient Range Scans: Adjacent keys stay together; scan is fast and sequential
  • Concatenated Indexes: Multi-column compound keys support efficient queries on remaining columns
  • Direct Access: Know partition boundaries → query one partition directly
  • Hot Key Mitigation Possible: Can split boundaries dynamically (at cost of complexity)

❌ Cons

  • Hot Spots from Monotonic Keys: Timestamp-based partitioning concentrates all writes to "today"
  • Uneven Data Distribution: Some ranges may naturally have more data (alphabetically or otherwise)
  • Rebalancing Complexity: Splitting/merging ranges requires data movement and coordination
  • Partition Boundary Management: Manual or automatic, requires monitoring

Canonical Example: Sensor Network with Compound Keys

USE CASE Primary key = (sensor_name, timestamp). Hash sensor_name for partition, sort by timestamp within partition. Query: "All readings from sensor X in range [T1, T2]" → single partition, sequential scan.

The Timestamp Hot Spot Problem

🔥
Hot Spot Risk
All writes go to current time partition. Other partitions (past dates) sit idle. Single partition becomes bottleneck.
✅
Mitigation: Prefix Key
Prepend sensor name or random value before timestamp: (sensor_name, timestamp). Distributes writes across sensors. Trade-off: multi-sensor range queries now require scatter/gather.
💡
Real-World Trade-Off
Distribute write load over read efficiency. Sensor writes are frequent; aggregate reads are rare. Makes sense to sacrifice range scan for load distribution.

Systems Using Key-Range Partitioning

  • 📊 Bigtable (Google)Tablets assigned to ranges of row keys; automatic boundary management
  • 📊 HBaseRegions (Bigtable's equivalent) with configurable key ranges and dynamic splits/merges
  • 📊 RethinkDBAutomatic range-based sharding with rebalancing
  • 📊 MongoDB (pre-2.4)Used range-based sharding; later switched to hash

05 Hash Partitioning

Hash Function: Skewed Keys → Uniform Distribution

Bucket 7evenly distributed

Use a hash function to transform skewed keys into a uniform distribution across partitions. Partitions are defined by ranges of hash values (not key values).

Core Mechanism

1

Hash the Key

Apply hash function (e.g., MD5, Fowler-Noll-Vo) to convert key to fixed-size hash digest.

2

Partition from Hash

Partition = hash(key) mod N_partitions. Or: hash values divided into ranges; find range containing hash.

3

Direct Access

Query knows hash value → direct to partition. No need to search boundaries.

Hash Function Requirements

  • 🔴 NOT Cryptographically SecureSpeed is priority, not security. MD5 is acceptable; SHA-1 overkill.
  • 🔴 Consistent Across ProcessesSame key must hash identically across all nodes. Avoid Java Object.hashCode() or Ruby Object#hash (randomized per process).
  • 🟢 Common ChoicesMD5 (Cassandra, MongoDB), Fowler-Noll-Vo (Voldemort, Riak)

Advantages vs. Disadvantages

✅ Pros

  • Fair Distribution: Hash distributes keys fairly across partitions, reducing hot spots
  • Evenly Spaced Boundaries: Can use simple mod arithmetic for partition assignment
  • Reduced Skew: For uniform access patterns, hash shines
  • Deterministic Routing: Any node can compute partition from key

❌ Cons

  • No Range Queries: Adjacent keys scattered across all partitions — sort order lost
  • Scatter/Gather Required: Range query [K1, K2] must hit every partition and gather results
  • Poor for Sorted Data: Inefficient for queries like "all updates between T1 and T2"
  • Rebalancing Overhead: Adding/removing partitions requires re-hashing everything
RANGE QUERY PENALTY Hash partitioning on timestamp means "all events in Q2" requires querying all N partitions in parallel, waiting for all to respond. With key-range, it's a single partition scan.

Consistent Hashing (Historical Note)

📚
Academic Definition
Karger et al. (1997): Randomly chosen partition boundaries placed on a hash ring. Minimizes data movement when nodes join/leave.
⚠️
Major Confusion
"Consistent" ≠ ACID consistency. Refers to rebalancing consistency (minimizing movement). Rarely used in practice — works poorly for databases.
