02 ACID Framework Overview

ACID Properties & Their Responsibilities
A: Atomicity All-or-nothing fault tolerance (crash recovery)
→
Database Responsibility. Ensures abortability via write-ahead log.
C: Consistency Application-level data invariants (foreign keys, balances)
→
Application Responsibility. Database cannot enforce user-defined logic.
I: Isolation Concurrent txns don't interfere (serializability)
→
Database Responsibility. Via locks, snapshots, or conflict detection.
D: Durability Committed data survives hardware failure
→
Database + Infrastructure Responsibility. Replication + backups.
HISTORICAL NOTE ACID was formalized in 1983 by Härder & Reuter. The "C" was, per Joe Hellerstein, "tossed in to make the acronym work" and considered less foundational than A, I, D. Today, ACID is largely a marketing term — implementations vary wildly.
📜
Acronym Overload
ACID and BASE are both marketing terms. Consistency means ≥4 different things (replica consistency, CAP consistency, ACID invariants, consistent hashing). Beware!
⚠️
False Marketing
NoSQL movement claimed "ACID = anti-scalability." False: ACID is orthogonal to scalability. The real trade-off is serializability vs. performance.
✅
Proven Pattern
IBM System R established the transaction pattern in 1975. ~40 years later, MySQL InnoDB, PostgreSQL, Oracle, SQL Server all follow essentially the same model.

03 Atomicity: Fault Tolerance via All-or-Nothing

Atomicity in Action: Crash Recovery
Scenario Without Atomicity (No WAL) With Atomicity (WAL)
Write 1 succeeds ✘ Partial state persists ✔ Logged, waits for Write 2
Write 2 starts, then CRASH ✘ Corrupted data after restart ✔ Rollback both writes, clean state
After recovery, application retries ✘ No idempotency guarantee ✔ Safe to retry: either all or nothing

Key Definition: Abortability

Atomicity is fundamentally about abortability. A transaction either completes entirely or leaves no trace. The term "atomic" is a misnomer (confusing with physics); "abortable" would be clearer.

🏗️ How Atomicity is Implemented

Write-Ahead Log (WAL): Before any write is applied to the database, it is logged to durable storage. On crash, the recovery process scans the log and either completes pending transactions (if marked committed) or rolls them back (if still in-progress). This guarantees that after recovery, the database is in a consistent state—either the transaction's writes are all present or all absent.

  • 🔵 Write-Ahead LogMechanism where database writes to durable log before applying changes. On crash, uncommitted log entries are discarded; committed entries are replayed.
  • 🟢 Safe Retry GuaranteeApplication can safely retry a failed transaction. Either it completes as if nothing happened, or it aborts cleanly with no partial state.
  • 🔴 Crash Recovery OverheadMultiple disk writes per transaction (log + data pages). SSD firmware bugs and sudden power loss can still cause subtle corruption despite WAL.

04 Consistency: Application Responsibility

⚠️
ACID Consistency
Application-level invariants (e.g., "all bookings must have valid room_id"). NOT a database property.
🔄
Replica Consistency
Data eventually syncs across replicas (eventual consistency). Orthogonal to ACID C.
🎯
CAP Consistency
Linearizability — all nodes see the same value simultaneously. Requires synchronous writes.
🔗
Consistent Hashing
Distributed hash ring for cache distribution. Completely unrelated to ACID consistency.

What Database Can Enforce

Foreign Key Constraints ✅ Database
Uniqueness (PRIMARY KEY, UNIQUE) ✅ Database
NOT NULL Checks ✅ Database
Custom Business Logic (e.g., "balance ≥ 0") ❌ Application
Complex Invariants (multi-table checks) ❌ Application
CRITICAL INSIGHT The "C" in ACID is NOT a database guarantee. It is application responsibility. The database provides only Atomicity and Isolation (A and I). Consistency depends on the application using them correctly.

05 Isolation: Concurrency Control & Serializability

The Race Condition Problem

Definition: Serializability

Transactions execute concurrently but produce results identical to serial execution (one after another). This is the gold standard of isolation and the formal definition of "isolated transactions."

Reality Gap: Rarely Used in Practice

Serializability is expensive (locks block readers and writers). Most databases offer weaker isolation levels to improve performance. Even worse: marketing deception — Oracle 11g advertises "serializable" mode but actually implements snapshot isolation (much weaker). This creates enormous confusion.

🎯 Ideal: Serializability

Results identical to serial execution. Prevents all anomalies (lost updates, phantom reads, write skew).

Safe ✔ Slow ✗ Rarely used ✗

⚡ Reality: Weak Isolation

Snapshot isolation, read committed, etc. Faster but allows anomalies. Most production systems use this.

Fast ✔ Risky ⚠ Widely used ✔
  • 🔵 Race ConditionTwo concurrent transactions read the same value, both increment, both write back. One increment is lost.
  • 🔴 Dirty ReadTransaction A reads uncommitted (in-progress) data from Transaction B. If B aborts, A has garbage.
  • 🔴 Phantom ReadTransaction A's WHERE clause returns different results mid-transaction because B inserted matching rows.
  • 🟠 Write SkewTwo transactions read the same data, check an invariant, then write different columns. Both pass the check but invariant is violated.

06 Durability: Storage, Replication & Risk

Durability Implementation Strategies
Single-Node: Write to Disk
Write-Ahead Log to HDD/SSD
  • ✔ Data survives machine crash
  • ✔ Data inaccessible until recovery
  • ✘ Correlated faults (power loss, bugs)
  • ✘ SSD data loss (power cut, weeks)
Multi-Node: Replication
Replicate to N nodes before ACK
  • ✔ Data accessible if one node dies
  • ✔ Mitigates correlated faults
  • ✘ Network partitions delay writes
  • ✘ All replicas could fail together

The Hard Truth: Perfect Durability is Impossible

If every disk is destroyed AND every backup location is destroyed, data is gone. Durability is a probabilistic guarantee: you can only reduce risk, not eliminate it.

✅ Replication Wins

  • Survives single-node hardware failure
  • Avoids cascading failures from correlated faults
  • Mitigates SSD power-loss failures
  • Data remains accessible during recovery

❌ Disk-Write Alone

  • All replicas could crash from power outage
  • fsync() not guaranteed during sudden power cut
  • Disk firmware bugs corrupt files post-crash
  • Data inaccessible during recovery window
SSD FAILURES SSDs develop ≥1 bad block in 30–80% of drives within 4 years. Magnetic drives have lower bad-sector rates but higher complete failure rates. Neither is reliable alone.
💡
Durability is Layered: Write-ahead log (single-node) + replication (multi-node) + backups (offsite) = reducing risk from multiple angles simultaneously.

07 Single-Object Operations & Lightweight Atomicity

What Atomicity Prevents (Single Object)
Problem Without Atomic Read With Atomic Read
Incomplete JSON read ✘ Partial old + new mixed ✔ Full old or full new
Concurrent write mid-read ✘ Inconsistent snapshot ✔ Reader blocked until write done
Increment operation ✘ Lost updates (race) ✔ Atomic read-modify-write

Built-in Atomic Operations (Database-Level)

  • 🟢 INCREMENTAtomic fetch-add. Multiple threads can safely increment; results are serialized.
  • 🟢 Compare-and-Set (CAS)Read value, compute new value, write only if unchanged. Prevents lost updates in single-threaded retry loops.
  • 🟢 Read-Modify-Write EliminationDatabase holds lock on object; client reads, computes, writes without releasing lock. No race window.
MARKETING INFLATION Vendors call single-object CAS operations "lightweight transactions" or claim "ACID on single objects." Misleading. A transaction groups multiple objects; single-object atomicity is just basic database hygiene, not transactions.

Why Multi-Object Operations Matter

Single-object atomicity is table stakes. Real transactions span multiple objects: foreign key references, denormalized documents, cross-table invariants. This is where transactions earn their value.

🔒 Implementation: Lock Per Object

Database maintains one lock per object. Before reading, client acquires a shared (read) lock; before writing, an exclusive (write) lock. If another transaction holds a conflicting lock, the reader/writer waits. On commit/abort, all locks release. This guarantees atomicity + isolation for single-object operations.

08 Multi-Object Transactions: The Real Purpose

1

Foreign Key Constraint

User table references account table. If you insert a user, its account_id must exist in accounts. Both writes must succeed together (atomic) or abort together.

2

Document Denormalization

NoSQL stores all order items inside a single order document (denormalized). When updating item list, entire document must be rewritten atomically. No joins, but requires atomicity.

3

Multi-Step Business Logic

Transfer $100 from Alice to Bob: decrement Alice's balance AND increment Bob's balance. Both must succeed; if one fails, both abort. This is the canonical example.

4

Coordinated Updates

Update booking status AND reserve room seat AND deduct payment. All succeed together or all rollback. No partial states visible to other transactions.

WITHOUT MULTI-OBJECT TRANSACTIONS You must manually handle failures: "Write to A failed, but B succeeded. Now what?" Compensation logic, retries, idempotency checks — all become application responsibility. Transactions eliminate this complexity.
📋
Relational Model Benefit
Foreign keys across tables guarantee referential integrity. Transactions ensure all related inserts/updates are atomic.
⚠️
Denormalization Trade-off
Document stores lack joins, so developers denormalize (copy data). Transactions handle the complexity of updating all copies atomically.
✨
Simplifies Application Code
Instead of error-handling cascades, you can write: "execute these writes; if any fail, abort all." Database guarantees consistency.

09 Serial Execution: The Simplest Isolation

Serial Execution Architecture
Transaction 1(full execution, single thread)
→
Transaction 2(waits for Txn 1 to commit)
→
Transaction N(queued transactions)

Core Principle: One Thread, No Concurrency

All transactions execute in a single-threaded loop. Each transaction runs to completion (commit or abort) before the next one starts. No locks, no race conditions, no deadlocks—serializability by design.

Why It Works

  • 🟢 Guarantees SerializabilityBy definition, single-threaded execution produces serial order. No concurrency anomalies possible.
  • 🟢 No DeadlocksNo locks = no deadlock detection, no wasted retries.
