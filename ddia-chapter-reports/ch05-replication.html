02Three Core Replication Paradigms

  • 🔵 Single-Leader (Master-Slave, Active-Passive) One designated leader accepts writes; followers replicate asynchronously or synchronously. Simplest model, easiest to reason about. Dominated in practice: PostgreSQL (9.0+), MySQL, Oracle Data Guard, MongoDB, RethinkDB.
  • 🟠 Multi-Leader (Master-Master, Active-Active) Multiple nodes accept writes concurrently. More complex conflict resolution required. Enables continued writes during leader failure. Use cases: data centers (inter-DC replication), offline-first applications.
  • 🟢 Leaderless (Dynamo-style) All nodes equivalent; clients write to multiple nodes concurrently. Gossip dissemination and read-repair. Highest write availability. Implementation: Amazon Dynamo, Riak, Cassandra. Requires client-side or server-side merging of siblings.
Historical Context Core principles unchanged since 1970s. Fundamental network constraints remain invariant: partial failures, network delays, asynchrony cannot be assumed away.

Core Components

  • 🔵 Leader Node Accepts all client writes, stores data locally, maintains replication log (sequence of write operations, also called change stream). Responsibility: serialize writes, assign sequence numbers.
  • 🟢 Follower Nodes Receive replication log from leader, apply writes in same order as leader, read-only from client perspective. Clients may read stale data if follower hasn't caught up.
  • 🟣 Replication Log (Change Stream) Sequence of write operations transmitted from leader to followers. Format depends on implementation method: statement-based, WAL-shipped, or logical (row-based).
  • 🔴 Consistency Invariant Critical guarantee: followers must apply writes in exact same order as leader. Deviation = data corruption across replicas.

04Synchronous vs. Asynchronous Replication: The Fundamental Trade-off

Dimension Synchronous Asynchronous
Durability Guarantee ✔ Write must be replicated to ≥2 nodes before acknowledgment ✘ Write reported to client before replication begins
Latency ✘ High — blocks on follower response (typically 50–100ms per follower) ✔ Low — acknowledgment immediate, replication decoupled
Availability ✘ Low — single follower failure halts writes at leader ✔ High — leader continues accepting writes regardless of follower state
Typical Lag < 1 second (usually sub-100ms) Can reach minutes, hours, or indefinitely (network partition)
Practical Use Rare in practice; expensive cost-benefit Default in most production systems (PostgreSQL, MySQL)

✅ Why Async Wins

  • Write latency decoupled from follower state. Leader not blocked by slow replicas.
  • Fault tolerance of leader unaffected by replica failures. System continues accepting writes.
  • Network robustness handles high-latency or unreliable inter-node links.
  • Practical reality in all major databases: PostgreSQL (asynchronous default), MySQL (asynchronous), MongoDB (asynchronous).

❌ The Async Weakness

  • Durability is weak. Leader crash may lose unreplicated writes. Violates ACID durability expectation.
  • Replication lag unpredictable. Reads from followers return stale data. Violates read-after-write consistency.
  • Failover hazard new leader may lack writes from old leader. Silent data loss.
  • Primary key reuse disaster possible if new leader autoincrement counter lags (GitHub incident: exposed private data).

Semi-Synchronous: The Pragmatic Middle Ground

🏆 Recommended Configuration

One follower synchronous, others asynchronous. Guarantees durability on ≥2 nodes (leader + 1 sync follower up-to-date) while preserving write availability if any single follower fails. Most practitioners' choice: trade slight latency increase for durability guarantee without sacrificing availability.

Critical Trade-off Cannot simultaneously achieve strong durability (sync), low latency, AND high availability. Must sacrifice at least one. Semi-synchronous sacrifices a small amount of latency for guaranteed 2-node durability + availability.

05Setting Up New Followers: Zero-Downtime Process

1

Take Consistent Snapshot of Leader

Obtain exact point-in-time snapshot without full database lock (most modern databases support this: PostgreSQL COPY, MySQL mysqldump --single-transaction). Captures all committed data at snapshot moment. Snapshot must be accompanied by exact replication position.

2

Copy Snapshot to New Follower

Bulk transfer snapshot data to new follower node (disk, network, cloud storage). Can happen gradually; no downtime required. New follower loads data into local storage structure.

3

Follower Connects and Requests Change Stream

New follower initiates connection to leader, sends replication position from snapshot (e.g., PostgreSQL log sequence number (LSN) or MySQL binlog file + byte offset). Requests all changes since that position.

4

Follower Catches Up

Processes change stream in sequence, applying writes to local storage in same order as leader. Once lag reaches zero, follower is fully replicated and ready to serve reads.

Position Tracking: Key Implementation Details

  • 🔵 PostgreSQL uses log sequence number (LSN)—a byte position in WAL (write-ahead log). Example: 0/3D5C370
  • 🟣 MySQL uses binlog coordinates—combination of binlog file name and byte offset. Example: mysql-bin.000003:154
  • 🟢 Position invariant replication position must be deterministic and tracked consistently to ensure followers don't miss or re-process changes.

06Node Outages: Failure Detection & Recovery

Follower Failure: Catch-up Recovery (Simple)

Follower Failure & Recovery
Follower maintains local log of received changes
On restart → connect to leader → "send me all changes since position X" → catch up asynchronously
Result: Automatic recovery, no manual intervention required.
  • 🟢 Straightforward mechanism Follower persists replication position after applying changes. On restart, queries leader for changes since that position.
  • 🟢 No data loss Follower has its own durable log; missed changes can be fetched from leader's replication log (kept for days/weeks).
  • 🟠 Replication lag during downtime If follower was down for extended period, catch-up may take time. Reads from follower return stale data until caught up.

Leader Failure: Failover (Complex)

Leader failure is the most dangerous scenario in replication. Automatic failover can trigger cascading failures, data loss, and corruption.

1

Detect Leader Failure

Timeout mechanism (typically ~30 seconds) detects unresponsiveness. Challenge: no reliable failure detection over networks. Slow network = timeout. Network partition = ambiguity (is leader dead or unreachable?).

2

Choose New Leader

Election among replicas (majority consensus) OR appointment by controller node. Candidate selection: replica with most up-to-date changes (least data loss). Position used: latest replication position in replica's log.

3

Reconfigure System

Clients rerouted to new leader (DNS, load balancer config, application config). Old leader becomes follower. Critical step: old leader must recognize its demotion and not continue accepting writes (prevents split-brain).

07Failover Failure Modes: When Automated Recovery Goes Wrong

💾
Unreplicated Write Loss
Scenario: Async replication, leader crashes before follower receives latest writes. New leader elected lacks those writes. Result: Silent data loss, violates durability expectations. Common solution: discard unreplicated writes (trade durability for availability).
🧠
Split-Brain (Two Leaders)
Scenario: Network partition isolates old leader. Other replicas elect new leader. Both old and new accept writes independently. Consequence: data divergence, requires complex merge logic to recover. Prevention: fencing/STONITH mechanism—forcibly shut down old leader to prevent writes. Risk: poor fencing design causes both to be shut down (all writes lost).
🔑
Primary Key Reuse Disaster (GitHub Incident)
GitHub case: Out-of-date follower promoted to new leader. Autoincrement counter lags old leader's. Same PKs assigned by new leader were previously assigned by old leader. Caused unique constraint violations, data corruption. Secondary damage: MySQL-Redis inconsistency exposed private data. Lesson: ensure new leader is most up-to-date candidate before promotion.
⏱️
Timeout Dilemma
Trade-off: Long timeout (e.g., 5 min) = slow recovery from actual failure. Short timeout (e.g., 10 sec) = unnecessary failovers under heavy load/network congestion, amplifying problems. Solution: monitor-based alerting + manual failover (many operators prefer this despite automatic support).
Operational Reality Many teams deliberately disable automatic failover despite database support. Manual failover is safer: allows human judgment, DNS propagation time, system state verification before promotion. Trades RTO (recovery time objective) for lower risk of cascading failure.

1. Statement-Based Replication

Log every SQL statement (INSERT, UPDATE, DELETE). Follower parses and executes statements.

✅ Advantages

  • Compact log (SQL text is small).
  • Human-readable for debugging.
  • Version-agnostic (same SQL runs on different versions).

❌ Critical Problems

  • NOW(), RAND() nondeterministic—each replica generates different values.
  • Autoincrement, conditional UPDATEs order-dependent; must execute identically on all replicas.
  • Triggers, stored procedures, UDFs side effects diverge per replica.
  • 🔴 Workaround complexity Leader must replace nondeterministic calls with fixed values before logging (e.g., NOW() → 2024-01-15 10:30:45). Adds implementation burden.
  • 🔴 Limited adoption MySQL <5.1 used this; now fallback only. VoltDB uses statement-based but requires deterministic transactions (restricted programming model).

2. Write-Ahead Log (WAL) Shipping

Concept: Storage engines already append all writes to WAL (used for crash recovery, main source of truth for mutations). Ship exact log bytes across network; follower rebuilds storage structure.

  • 🔵 Core idea WAL is already maintained by storage engine (B-tree, LSM) for durability. Replication reuses this log—efficient, no translation needed.
  • 🔴 Storage engine coupling WAL describes low-level byte changes per disk block, tightly coupled to storage format (B-tree node layout, LSM memtable structure). Version mismatch = replication incompatibility.
  • 🔴 Zero-downtime upgrades blocked Cannot upgrade follower to newer software version (different WAL format). Forces full outage + snapshot + resync. Operational nightmare in large systems.
PostgreSQL, Oracle Limitation Both use WAL shipping as primary replication method. Cannot perform zero-downtime rolling upgrades of replication followers (major operational constraint).

3. Logical (Row-Based) Replication

Encode row changes in semantic format decoupled from storage engine. Follower interprets changes based on logical meaning, not physical storage layout.

  • 🟢 Version flexibility Semantic format allows follower to use different storage engine version. Follower can upgrade independently. Enables zero-downtime rolling upgrades (promote replica, upgrade leader, re-replicate).
  • 🟢 Semantic clarity Log describes rows affected, not bytes modified. Easier to understand, debug, audit mutations.
  • 🟠 Slightly larger log Encodes row IDs, column values, not just byte offsets. Storage overhead moderate (still compact compared to statements).

09Concurrency, Happens-Before, and Defining Consistency

Happens-Before Relation (Event Ordering)
Operation A happens-before B if:
✔ B knows about A (received its result)
✔ B depends on A (needs A's output)
✔ B builds upon A (chains causally)

Concurrency definition: A and B concurrent ⟺ neither happens-before the other (neither knows about the other)

Trichotomy: For any two ops A, B: (1) A → B, OR (2) B → A, OR (3) A ∥ B

Key Insight: Physical Time is Irrelevant

Exact wall-clock time doesn't define concurrency. Two operations may have a time gap but still be concurrent if network prevented mutual awareness.

  • 🔵 Relativity principle Connected to special relativity: information cannot travel faster than light. Two distant events cannot affect each other if time between them is shorter than light travel time. Network analog: two nodes separated by latency cannot coordinate if message round-trip exceeds operation gap.
  • 🟣 Network caveat Two operations can be concurrent despite physical time gap if network slowness or partition prevented one from knowing about the other. Clock skew doesn't matter; causality matters.
  • 🟢 Practical consequence Don't use wall-clock timestamps to order events in distributed systems. Use logical clocks (version numbers, vector clocks) to capture causal relationships.

💡 Why This Matters for Replication

Concurrency definition based on causality (not physical time) enables leaderless systems. Nodes don't need synchronized clocks to detect which writes conflict. Version vectors capture concurrency precisely.
