Three Mutually Exclusive Replication Algorithms

Single-Leader Replication — One node accepts writes Most Common
Multi-Leader Replication — Multiple writers, complex conflicts Moderate
Leaderless Replication — All nodes equal, client-driven conflicts Specialized

Core Challenge: Replication is about handling changes to data, not managing static snapshots. Every write must propagate reliably and consistently across all nodes.

Architecture Components

  • 🔵 Leader (Master/Primary) Sole node accepting write operations; persists to local storage first; immediately generates change events
  • 🟢 Followers (Read Replicas/Secondaries) Receive serialized stream of writes from leader; apply changes identically in same order; clients read from leader OR followers
  • 🟠 Replication Log Sequential record of every write operation; followers consume and replay locally; forms backbone of consistency
  • 🟣 Client Interface All writes directed to leader; reads optionally distributed across followers for load balancing

KEY INSIGHT Single-leader replication's simplicity and prevalence stem from its straightforward write ordering: every follower replays the exact same operations in identical sequence, eliminating complex conflict resolution.

04 Synchronous vs Asynchronous Replication

🔵 Synchronous

Behavior: Leader blocks write acknowledgment until follower confirms receipt.

✓ Durability ✗ Latency
⚡ DEFAULT

🟢 Semi-Synchronous

Behavior: One follower synchronous, rest asynchronous. Optimal balance.

✓ Durable ✓ Available

🟠 Asynchronous

Behavior: Leader acknowledges immediately; followers catch up eventually.

✓ Throughput ✗ Durability

Detailed Comparison

Mode Guarantee Advantage Disadvantage Use Case
Synchronous ✔ Durability Follower guaranteed up-to-date; data safe if leader fails Any follower outage blocks all writes; impractical with many followers Critical systems requiring zero data loss
Asynchronous ✘ None Writes continue even if all followers down; maximum throughput Durability guarantee lost; data lost if leader fails before replication High-volume analytics; non-critical systems; distributed followers
Semi-Sync ⚠ Partial ≥2 nodes have up-to-date copy (leader + 1 sync follower); automatic promotion on failure Promotion latency; limited synchronous follower availability Production databases (PostgreSQL, MySQL default); balance of safety & performance

🏆 Industry Practice: Asynchronous Dominates

Despite durability risks, asynchronous replication is widely deployed, especially with geographically distributed followers. Organizations accept eventual consistency trade-off for operational simplicity and write availability.

Semi-Synchronous Promotion Strategy

1

Normal Operation

Follower A designated synchronous; Followers B, C, D asynchronous. Every write requires A's acknowledgment.

2

Follower A Becomes Unavailable

Leader detects timeout; system cannot proceed with A's confirmation.

3

Automatic Promotion

Promote async follower (e.g., B) to synchronous role. Continue accepting writes with B's confirmation.

4

A Returns Online

A reconnects, catches up via replication log, potentially demoted back to async.

05 Setting Up New Followers (Zero-Downtime)

1

Take Consistent Snapshot (No Locking)

Leader creates snapshot without locking entire database. Most databases support this natively (required for backups anyway). Tools like innobackupex for MySQL automate this.

2

Copy Snapshot to New Follower

Transfer snapshot file(s) to new follower node over network or storage system. No special ordering required.

3

Connect to Leader & Request Backlog

New follower connects to leader, provides exact position in replication log (snapshot's point). Leader streams all changes since that position.

4

Replay & Reach Consistency

Follower processes backlog of changes, applies them in order, reaches consistency with leader, continues streaming new writes.

Log Position Naming by System

  • 🐘 PostgreSQL — log sequence number (LSN) identifies exact position in write-ahead log
  • 🐬 MySQL — binlog coordinates (filename + offset) pinpoint exact position in binary log
  • 🟠 Generic Pattern — Any replication system must encode snapshot position in language the leader understands
ZERO-DOWNTIME ADVANTAGE Snapshot does not lock database; new followers can be added without interrupting read/write traffic. This is critical for production systems scaling read capacity.

06 Failover: Hazards & Complexities

Three Automatic Failover Steps

1

Detect Leader Failure

Monitor uses timeout on heartbeat/message exchange (e.g., 30 seconds). Problem: Timeouts are imprecise—crashes, power failures, and network partition all look identical. No foolproof detection exists.

2

Choose New Leader

Election by replica majority OR appointment by controller node. Best candidate: replica with most up-to-date data (minimizes data loss). Consensus problem (see Ch9).

3

Reconfigure System

Update clients to route writes to new leader. Force old leader into follower role (becomes read-only). Prevents accidental dual-leadership.

Failover Hazards (No Easy Solutions)

💾
Data Loss
Problem: New leader may lack writes old leader issued before crashing; asynchronous replication loses unreplicated data. Example: GitHub incident—out-of-date MySQL follower promoted to leader; autoincrement counter reused primary keys previously assigned, exposing private data.
🧠
Split Brain
Problem: Two nodes both believe they're leader; both accept writes → data loss/corruption. Mitigation: Fencing mechanism shuts down node if 2 leaders detected, but poorly designed systems shut down both.
⏱️
Timeout Calibration
Dilemma: Long timeout = slow recovery; short timeout = unnecessary failovers during load spikes or transient network glitches, worsening the situation.
INDUSTRY PRACTICE Many teams disable automatic failover and prefer manual intervention. Reduces surprise data loss, but requires operational response time. Trade-off: availability vs. safety.

Failover Risk Comparison

✅ Automatic Failover

  • Fast recovery from leader failure
  • High availability for read-heavy workloads
  • Minimal operational effort

❌ Manual Failover

  • Operator must detect & diagnose failure
  • Slow promotion of new leader
  • Prevents write availability during outage

07 Replication Log Implementation Methods

📝
Statement-Based
Forward every SQL statement (INSERT, UPDATE, DELETE) to followers for replay
🔤
Write-Ahead Log (WAL)
Low-level log of disk block changes; followers replicate exact bytes modified
📦
Logical Log
High-level record of row-level changes; decoupled from storage format
🔔
Change Data Capture
Application observes database changes as events; external streaming system consumes

Statement-Based Replication: Failures

Failure Mode Description Example
Nondeterministic Functions NOW(), RAND() produce different values per replica INSERT INTO events (timestamp) VALUES (NOW()) — each replica records different time
Concurrency Dependency Autoincrement, conditional updates must execute in identical order across all nodes UPDATE table SET id=NEXT_ID() WHERE ... — order matters; failures cascade
Side Effects Triggers, stored procedures, UDFs may produce different side effects if execution order differs Trigger fires during statement; replica's trigger fires at different logical time → inconsistency

Write-Ahead Log (WAL) Replication

✅ Advantages

  • Exact byte-level replication; no ambiguity
  • Handles all statement types, including nondeterministic functions
  • Used by PostgreSQL, Oracle

❌ Disadvantages

  • Tightly couples replication to storage engine format
  • Followers cannot use different storage engine versions
  • Large log size (low-level details)

Logical Log (Row-Based) Replication

✅ Advantages

  • Decoupled from storage engine; more portable
  • Followers can run different versions, even different databases
  • External tools (Kafka, CDC systems) can consume for analytics

❌ Disadvantages

  • More verbose than WAL for large bulk operations
  • Requires row-level granularity (not suitable for column-based stores)
TREND Modern systems favor logical logs (MySQL binlog, PostgreSQL logical decoding) for flexibility. WAL remains dominant for single-node performance; multi-leader scenarios prefer logical isolation.

Replication Flow Breakdown

  1. Client sends WRITE request to leader node (only destination for writes)
  2. Leader processes & persists to local storage (durability checkpoint)
  3. Leader generates log record containing the change (statement, WAL block, or row delta)
  4. Log record streams asynchronously/synchronously to all followers (depending on sync mode)
  5. Each follower applies the log record to its local storage in identical order
  6. Followers acknowledge processing to leader (if synchronous/semi-sync mode)
  7. Leader acknowledges to client (once durability requirements met)
  8. Subsequent READs can target any follower (read scaling benefit)
KEY PROPERTY Followers apply the same log in the same order. This deterministic replay ensures replicas remain consistent even if followers are temporarily slow or disconnected (catch-up recovery).

09 Concurrency & Happens-Before Relationship

📊
Happens-Before Definition
Operation A happens before B if B knows about A, depends on A, or builds upon A's result
⚡
Concurrency Definition
Two operations concurrent if neither happens before the other (neither knows about each other)
🌐
Three-Way Partition
For any two operations A, B: either A→B (A before B), B→A (B before A), or A||B (concurrent)
🚀
Relativity Twist
Network delays can make operations concurrent even if physical timing would theoretically allow causality

Physics Analogy: Special Relativity

🌌 Information Cannot Travel Faster Than Light

Two events at distance D apart cannot affect each other if time between events < D ÷ speed of light. Distributed systems apply same principle: exact physical time is irrelevant for defining concurrency; only what each operation knows matters.

Happens-Before Relation Properties

  • 🔵 Transitive If A→B and B→C, then A→C (chain of causality)
  • 🔵 Partial Order Not all pairs (A,B) are ordered; some remain concurrent
  • 🔵 Knowledge-Based Defined by what information flows, not wall-clock time
  • 🔵 Causal Consistency Systems respecting happens-before prevent causality violations
IMPACT Happens-before formalizes intuition: two operations are concurrent iff neither can influence the other. This is the foundation for detecting conflicts in multi-leader & leaderless systems.

10 Version-Based Concurrency Detection Algorithm

Single-Replica Version Number Scheme

🔢
Data Structure
Server maintains integer version number for every key; incremented with each write to that key; stored alongside value
📖
Client-Server Protocol
READ: Server returns all non-overwritten values + latest version. WRITE: Client sends new value + version from prior read. Server overwrites values ≤ sent version; keeps higher versions as siblings

Client-Server Protocol Detailed

Step Action Key Detail
Read Client requests key from server Server returns all non-overwritten values plus latest version number. Client must read before write.
Write Prep Client merges all received values Client combines siblings from read into single value (application-specific merge logic)
Write Client sends new value + version from read Must include version number received in prior read; identifies what client "knows about"
Server Processing Overwrite values at sent version and below Sent version marks boundary: ≤ sent version = client saw these, so overwrite. > sent version = concurrent with incoming write, keep as sibling

Shopping Cart Example (Figure 5-13 Walkthrough)

5 writes, 2 clients, single replica, building shopping cart:

1

Client 1: Add Milk

Client 1 adds milk → Server: {milk}@v1. Version 1 created. State: single item.

2

Client 2: Add Eggs (Unaware of Milk)

Client 2 reads cart (empty from its perspective), adds eggs → Server: {milk}@v1, {eggs}@v2. v2 is concurrent with v1; both coexist.

3

Client 1: Add Flour (Merge Milk)

Client 1 reads {milk}@v1, adds flour → sends {milk,flour}@v1. Server overwrites v1 (milk), keeps v2 (eggs) as sibling → {milk,flour}@v3, {eggs}@v2.

4

Client 2: Merge Received Values + Add Ham

Client 2 reads {milk}@v1, {eggs}@v2, merges to {milk,eggs}, adds ham → sends {eggs,milk,ham}@v2. Server overwrites v2, concurrent with v3 → {milk,flour}@v3, {eggs,milk,ham}@v4.

5

Client 1: Merge Stale + Add Bacon

Client 1 merges {milk,flour}@v3 + {eggs} (stale read), adds bacon → sends {milk,flour,eggs,bacon}@v3. Overwrites v3, concurrent with v4 → {milk,flour,eggs,bacon}@v3, {eggs,milk,ham}@v4. Final: two siblings, no data lost.

CRITICAL INSIGHT Server distinguishes overwrites (version ≤ sent) from concurrent writes (version > sent) using version numbers alone. Does not interpret values. This enables automatic detection without application knowledge.

Write-Without-Version Behavior

If client sends write without version number: creates sibling concurrent with all existing versions; overwrites nothing. Leads to unbounded sibling explosion if not careful.

11 Conflict Resolution & Sibling Merging

The Merge Problem

⚠️
Siblings Definition
Concurrent values coexisting in system (Riak terminology). Equivalent to multi-leader conflict resolution problem.
💥
Data Loss Risk
Naive approaches (last-write-wins) discard concurrent writes. Shopping cart conflict resolution must preserve all items.
🔧
Application Burden
Merging is application-level; system cannot know correct resolution without domain knowledge

Union Merge Strategy (Add-Only Pattern)

✅ Union Merge for Additions

Strategy: Take union of all sibling values. Works perfectly for add-only data structures (shopping carts, tags, sets).

Example: {milk,flour} ∪ {eggs,milk,ham} = {milk,flour,eggs,ham} ✓ All items preserved.

❌ Union Merge Fails on Deletions

Problem: Removed items reappear when merging union of siblings where one version deleted item.

Example: v3={milk,flour,eggs,bacon}, v4={eggs,milk,ham}. If v4 had deleted flour, union recreates it—wrong!

Tombstone Solution for Deletions

🪦 Tombstones: Deletion Markers with Version Numbers

Mechanism: Item deletion is not erased; instead, a tombstone marker with appropriate version number is created. During sibling merge, tombstones prevent deleted items from reappearing.

Example: v4 deletes flour → {eggs, milk, ham, TOMBSTONE(flour@v4)}. Union preserves tombstone; flour stays deleted even if v3 contained it.

Automatic Conflict Resolution via CRDTs

🤖
CRDT (Conflict-free Replicated Data Type)
Data structures that automatically merge siblings sensibly without application code. Preserves deletions correctly; avoids complex error-prone manual merging.
📦
Riak Datatype Support
Riak uses CRDT family (LWW Register, OR-Set, Counter, Map) to handle common conflicts automatically
🧮
Common CRDT Types
OR-Set (add-remove sets), LWW-Register (last-write-wins), G-Counter (append-only), PN-Counter (increment/decrement)
MODERN PRACTICE CRDTs eliminate manual conflict resolution for standard data types. Applications using Riak datatypes or CRDT libraries avoid boilerplate merge logic and data loss risks.

12 Version Vectors for Multi-Replica Systems

Single Version Number Limitation

PROBLEM Shopping cart example used single replica. Single version number cannot capture dependencies when multiple replicas accept concurrent writes. Example: Client reads from Replica A (v5), then writes to Replica B. Replica B cannot determine if v5 from Replica A happened before or concurrent with its own version numbers.

Version Vector Structure

📊
Definition
Collection of version numbers, one per replica. Encodes causality across replica boundaries.
🔢
Per-Replica Version Tracking
Each replica increments its own version when processing writes. Each replica tracks version numbers seen from all other replicas.
🔗
Causality Encoding
Vector captures which writes have been seen; prevents causal ordering violations across replicas

Multi-Replica Protocol (Parallel to Single-Replica)

Step Action Version Vector Role
Read Client reads from any replica Server returns value + version vector (Riak encodes as "causal context" string)
Write Client sends value + version vector from read Any replica can accept write; vector identifies prior state; enables conflict detection across replicas
Replication Replica propagates write to other replicas Vector accompanies write; other replicas merge vector into their state; enables sibling detection
Merge Decision Replica compares vectors If vectors incomparable (neither dominates), writes are concurrent → create siblings

Key Property: Safe Cross-Replica Reads & Writes

✅ No Data Loss Across Replicas

Safe pattern: Read from Replica A, write to Replica B, read from Replica C, write to Replica A. Result: Siblings created if concurrent, but no data lost (assuming correct merging logic). Version vector ensures causal consistency regardless of replica routing.

Dotted Version Vectors (Riak 2.0)

  • 🟣 Riak 2.0 Innovation Most interesting variant of version vectors. Mechanically similar to shopping cart example but extended across replicas
  • 🟣 Details Deferred Conceptually extends version vectors with logical clocks; provides more precise causality tracking than traditional vectors
  • 🟣 Practical Impact Reduces false concurrent detection; minimizes sibling explosion in high-concurrency scenarios
VECTOR COMPARISON Version vectors enable pairwise comparison: A → B (A happened before B), B → A, or A||B (concurrent). This is the foundation for distinguishing overwrites from concurrent writes in leaderless systems (Dynamo, Riak, Cassandra).

13 Handling Node Outages & Recovery

Follower Failure: Catch-Up Recovery (Simple)

✅ Follower Catch-Up

Process: Each follower maintains local log of received changes. On restart/reconnection: replays missing changes from leader using stored transaction position.

Advantage: Simple because exact position in replication log is known (stored locally). Leader streams changes since that position; follower deterministically replays.

Example: Follower crashes at binlog position 500. On restart, reads "last known position: 500" from disk, requests changes 501+, replays automatically.

14 Key Components & Roles

  • 🔵 Leader/Master Single write authority; maintains replication log; propagates changes to all followers; point of failure for write availability
  • 🟢 Followers/Read Replicas Read-only; consume replication log; apply writes in order; serve read queries; elected as new leader on leader failure
  • 🟠 Replication Log Ordered sequence of write operations (statement, WAL, or logical); followers deterministically replay; enables catch-up recovery
  • 🟣 Version Number / Version Vector Per-key (single-replica) or per-replica (multi-replica) counter; distinguishes overwrites from concurrent writes; enables conflict detection without application knowledge
  • 🟣 Tombstone Marker Deletion record with version number; prevents deleted items from reappearing during sibling merge; core of correct CRDT deletion semantics
  • 🟣 CRDT (Conflict-Free Replicated Data Type) Data structure (OR-Set, LWW-Register, Counter) that automatically merges siblings correctly; eliminates manual conflict resolution
  • 🔴 Failover Detector Monitors leader health via heartbeat/timeout; initiates election on failure detection; prone to false positives
  • 🔴 Election Mechanism Majority-based or controller-appointed; selects new leader; consensus problem (Chapter 9); requires quorum to prevent split-brain
