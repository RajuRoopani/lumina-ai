Storage Engine Specialization
Each system optimizes for specific access patterns. B-trees excel at range queries; hash indexes fail on ranges. OLTP indexing differs from analytics compression.
⚠️
Naive Composition Risk
Allowing applications to write directly to multiple systems creates concurrent conflicting writes and ordering violations—permanent inconsistency if done unsafely.
🔗
CDC as Integration Spine
Change Data Capture treats single source-of-record as authority. Derived systems consume CDC stream in deterministic order, guaranteed consistency without atomic commit overhead.

Modern applications operate across multiple specialized storage engines:

  • OLTP transactional database — primary data store, consistency guarantees per transaction
  • Full-text search index — keyword queries, ranking, inverted index structure
  • Analytics data warehouse — columnar storage, aggregations, joins across years of data
  • Cache layer — in-memory key-value for read-heavy workloads, TTL eviction
  • Machine learning pipelines — feature engineering, training datasets, batch inference
  • Message queues — notifications, event stream processing, async workflows

The core tension: no single database solves all workloads efficiently. Forcing everything into one storage engine sacrifices performance, scalability, or operational simplicity.

Core Principle: Total Order Guarantee

A single system of record writes first. CDC (Change Data Capture) extracts changes and applies them to derived systems in identical order. This pattern guarantees consistency without any distributed transaction protocol:

1

Authoritative Write

All writes to persistent data hit the source database first. This is the single source of truth for all system state.

2

CDC Extracts Changes

Change Data Capture reads database logs (or transaction logs) and produces an ordered stream of mutations. MySQL binlog, PostgreSQL WAL, Oracle GoldenGate, Debezium—all guarantee order within a partition.

3

Deterministic Replay

All derived systems (search index, analytics, cache) consume CDC stream in same order. Assuming deterministic processing, all replicas converge to identical state.

4

Fault Recovery

If a derived system crashes, restart from last checkpoint and replay CDC events. Idempotent operations + determinism = automatic consistency recovery without coordination.

ANTI-PATTERN: Direct Multi-System Writes Allowing applications to write directly to both database and search index creates race conditions. Concurrent conflicting writes can be ordered differently in each system—permanent inconsistency. CDC solves this: single ordering point, all systems respect it.

04Log-Based Derivation vs Distributed Transactions

Property Distributed Transactions (2PC/XA) Log-Based Derived Data (CDC)
Ordering mechanism Locks (mutual exclusion, 2PL) Log ordering (total order broadcast)
Atomicity method Atomic commit protocol (coordinator + prepare/commit phases) Deterministic retry + idempotence
Consistency guarantee ✔ Linearizability (strong) Eventual consistency (async updates)
Read-your-own-writes ✔ Yes (by default) ✘ No (timing not guaranteed)
Throughput ceiling Severely limited by coordinator coordination ✔ Scales with parallelism
Fault tolerance Poor; can amplify failures across boundaries ✔ Better; failures isolated locally
Operational complexity High (distributed consensus, deadlock detection) ✔ Lower (replay, idempotence)

🏆 Author's Position

Log-based derivation is more promising than distributed transactions, despite weaker consistency guarantees. The goal: find middle ground—retaining safety without forcing "suck it up" eventual consistency on applications that need stronger guarantees.

Why 2PC Fails at Scale

  • Coordinator bottleneck — single decision point for all transactions; blocks all participants during prepare/commit phases
  • Availability trade-off — if coordinator unavailable, entire system blocks; no progress possible
  • Network partition amplification — one failed node can block dozens of other systems
  • Deadlock potential — nested locking across multiple systems increases deadlock probability
  • Limited by slowest participant — round-trip latency multiplied across network hops

Why Log-Based CDC Excels

  • Single ordering point — log guarantees order; no cross-system coordination needed during write
  • Decoupled systems — each derived system advances independently; one lagging doesn't block others
  • Idempotent operations — applying same change twice yields same result; enables safe replay after crashes
  • Deterministic processing — given same inputs in same order, all replicas produce identical state
  • Operational robustness — failures contained to one system; others continue; no cascading blocks
💡
Core Insight: Distributed transactions solve consistency synchronously at write time; log-based derivation achieves consistency asynchronously via deterministic replay. The latter scales dramatically better and tolerates failures more gracefully.

05Capturing Causality in Distributed Events

Social Network Causal Dependency Example
User unfriends ExEvent A
← happens first logically
User messages Remaining FriendsEvent B
← depends on A, mentions Ex-partner
🚨
Causal Ordering Risk
If notification service processes message-send before unfriend event, system incorrectly notifies ex-partner despite being unfriended. Causal dependency lost → permanent data corruption.
⚠️
Hidden Dependencies
Friendship stored separately from messages → causal link invisible to system. No ordering constraint in total-order broadcast prevents out-of-order delivery across independent streams.
🔍
Research Gap
No standard mechanism efficiently captures causal dependencies without coordinating through single bottleneck. Logical timestamps proposed but require metadata propagation.

The Problem: Arbitrary Event Ordering

Without explicit causal links between events, any ordering is acceptable from system perspective. But causal dependencies arise subtly in application logic. User intent ("unfriend before notifying") differs from system interpretation of independent events.

Proposed Solutions

🔢 Logical Timestamps

Lamport clocks or vector clocks provide total ordering without coordination. Recipients handle out-of-order delivery via reordering or buffering.

No bottleneck Requires metadata Application complexity

📝 Event Logging + State Capture

Log event plus system state user saw before decision. Give event unique ID. Later events reference ID to record explicit causal dependency.

Explicit deps Storage overhead Still async

⚙️ Conflict Resolution

Algorithms help process out-of-order events, maintain correct state. Example: CRDT (Conflict-free Replicated Data Type) merges concurrent edits deterministically.

Handles divergence No external side effects Limited applicability

🔗 Partitioned Causality

Route all causally-related events to same partition. Single stream ordering guarantees causality. Trade-off: throughput vs. logical cohesion.

Simple to reason Potential bottleneck Proven pattern
OPEN PROBLEM Capturing causal dependencies efficiently without forcing total order bottleneck remains active research. Current best practice: identify critical causal paths; route through shared partitions; accept eventual consistency elsewhere.

Unifying Principle: Consume, Transform, Join, Aggregate, Write

Both batch and stream processing achieve the same goal: ensure data in right form, right places. Output types include:

  • Search indexes — inverted index of documents for keyword queries
  • Materialized views — precomputed aggregations, denormalized schemas
  • Recommendations — user-item similarity computed offline
  • Aggregate metrics — count, sum, quantiles for dashboards
  • ML model outputs — predictions, embeddings, feature importance

The Convergence Trend

Implementation boundaries blurring between batch and stream:

Spark Structured Streaming — microbatches on batch engine Stream layer
Apache Flink — batch processing on stream engine Unified approach
Performance gap — characteristics differ significantly Trade-off
💡
Key Insight: Stream = unbounded inputs; Batch = bounded, finite inputs. Core processing principles (deterministic operators, idempotence, checkpoint-recovery) apply to both. Choose engine based on latency requirements and data size, not fundamental capability.

07Architecture Patterns: Lambda, Kappa, and Beyond

Λ
Lambda Architecture
Speed layer (stream) for real-time; batch layer for accuracy. Combine both views at query time. Trade-off: complexity, operational overhead of dual pipelines.
Κ
Kappa Architecture
Single stream pipeline with replay capability. Simpler operationally—all logic in stream processor. Re-processing old data by replaying from beginning.
⚙️
ETL (Extract-Transform-Load)
Classical pattern: pull from sources, transform, load to warehouse. Periodic execution (nightly batch). Trade-off: freshness vs. processing cost.
∞
Continuous Integration
CDC + stream processing for continuous updates. Change flows immediately from source to derived systems. Lower latency, no batch windows.

When to Use Each Pattern

Pattern Latency Accuracy Operational Cost Complexity
Lambda Mixed (seconds + batch window) ✔ High (batch reconciles) High (2 pipelines) Very High (code duplication)
Kappa ✔ Low (stream) High (if deterministic) ✔ Low (single pipeline) Medium (replay logic)
ETL Batch High (periodic, delay) ✔ High (full scan) Medium (batch window) ✔ Low (well-understood)
CDC Stream ✔ Very low (event-driven) High (if idempotent) ✔ Medium (always-on) Medium (ordering logic)
MODERN BEST PRACTICE Favor Kappa or CDC-based architectures. Lambda's operational overhead (maintaining 2 codebases, reconciling divergences) rarely worth accuracy gain. Stream engines now mature enough for production use. Replay capability handles accuracy edge cases.
